## 06-08 — 시대 연혁 (E1 + E2)

### A. CSV 두 개 → DB 적재 (어르신 검증된 데이터)

**범위**: 1980~2019, 사건 88건 + 음악 73곡. 아카이빙 방에서 검증된 CSV.

**파일 구조**:
```
db/seed/era-events/
  ├── _generate.ts                       # CSV → .ts 자동 변환기 (RFC 4180 미니 파서)
  ├── lifebook_MonthEvent_1980-2019.csv  # 사건 원본
  ├── lifebook_music_1980-2019.csv       # 음악 원본
  ├── era-events.ts                       # AUTO-GENERATED (eraMonthEvents 88건)
  └── era-music.ts                        # AUTO-GENERATED (eraChartSongs 73곡)
db/
  ├── inspect-era-csv.ts                  # 일회성 진단 (88/73 매핑 100% 확인)
  └── seed-era-events.ts                  # 적재 스크립트 (deterministic id + upsert)
```

**RFC 4180 파서** — 곡명·가수에 쉼표 있는 행 6건 ("윤도현 밴드(YB), 붉은악마", "소유, 정기고", "눈, 코, 입" 등) 정상 보존. UTF-8 BOM 처리.

**enum 매핑 100%** — 정치사회→POLITICS_SOCIETY / 문화연예→CULTURE / 스포츠→SPORTS / 생활경제→TREND / 검증됨→VERIFIED / 추정→APPROX / 국내→DOMESTIC / 해외→INTERNATIONAL.

**deterministic id**:
- MonthEvent: `SHA-256(section|year|month|title).slice(0, 24)` — `seed-timemachine.ts` 와 **동일 함수 규칙** (두 시드가 같은 자연키 = 충돌 0)
- ChartSong: `SHA-256(origin|year|month|title|artist).slice(0, 24)`
- 재실행 idempotent — 한 번 더 돌려도 새로 생성 0 / 갱신만

**기존 데이터 보존** — `seed-timemachine.ts` 의 ChartSong 은 `deleteMany+createMany` 패턴이라 그대로 쓰면 2025-2026 음악 128곡이 날아감. 우리 새 스크립트는 ChartSong 도 per-row upsert.

**적재 결과**:
```
[적재 전]
  MonthEvent: 총 46건  (1980~2019: 0, 2025+: 46)
  ChartSong:  총 128건  (1980~2019: 0, 2025+: 110)

MonthEvent upsert: 새로 생성 88건, 갱신 0건
ChartSong  upsert: 새로 생성 73건, 갱신 0건

[적재 후]
  MonthEvent: 총 134건  (1980~2019: 88, 2025+: 46)  ✓ 보존
  ChartSong:  총 201건  (1980~2019: 73, 2025+: 110)  ✓ 보존
```

### C. E2 — 클릭 한 번으로 내 연혁에 담기

**핵심 동작**: /era 사건 카드에 "내 연혁에 담기" → 그 사건이 사용자의 인생 연혁(UserMemory)에 추가됨 → 연혁에 slate 톤 "시대 배경" 카드로 표시.

**DB 변경 — partial unique index 한 줄**:
- 마이그: `20260608000000_era_event_unique/migration.sql`
- ```sql
  CREATE UNIQUE INDEX "UserMemory_userId_monthEventId_createdVia_key"
    ON "UserMemory" ("userId", "monthEventId", "createdVia")
    WHERE "monthEventId" IS NOT NULL;
  ```
- partial WHERE 절은 Prisma `@@unique` 가 표현 못 함 → schema 에는 주석만, 마이그 SQL 만 partial. 코드는 P2002 catch 패턴 (compound where 헬퍼 안 씀)
- 적용 전 충돌 검증 (`db/check-era-unique-conflict.ts`): monthEventId 있는 행 = 0, 충돌 = 0. 안전 적용 확인 후 `prisma migrate deploy`
- 3컬럼 묶음 이유 — `timemachine_event` (월 화면 keep, 지금은 0건이지만 보존) 와 `era_event` 는 별도 라이프사이클. 같은 사용자가 같은 사건을 두 흐름으로 가질 수 있음 (분리 보존)
