# 결정 — 핵심 UX 를 "타임머신" 방식으로 피벗

## Problem

Phase 5 (출생연도 기준 타임라인) + 6 (음악 RAG 트리거) 으로 만든 기존 화면은
"내 인생 연혁" 을 한 페이지에 쭉 보여주는 구조였다. 사용자가 직접 만져보고
나서 드러난 약점:

- **회상 트리거가 약함** — 1979→2025 연도순 큰 리스트만 보여줌. 사용자가 "어느 달
  이야기를 적어야 하지?" 결정해야 하는 부담이 큼. 시니어에겐 자유도가 곧 막막함.
- **음악 추천이 흐림** — Voyage 임베딩 + 세대 가중치로 트리거를 끌어왔지만,
  "이 곡이 왜 추천됐는지" 가 불투명. 사용자 임의 곡 1개 = 시드 임의 곡 1개와의
  코사인 유사도라 추천 관계가 흐릿함.
- **저작권 위험 잠재** — 트리거 카드가 음악 정보(가수·곡명)만 보여주지만, 추천
  알고리즘이 확장되면 가사·미리듣기 같은 침범 가능 디자인으로 흐를 여지.

핵심 통찰: **회상은 "내가 살았던 시간의 절단면"이 가장 강력한 단서**. 1987년
6월 한 달의 풍경 = 그 달의 정치·사회·문화·스포츠·유행 + 그달 차트 음악이
한 페이지에 보일 때 회상이 가장 잘 일어남. "1979부터 보세요"가 아닌 "1987년
6월에 당신은 어땠어요" 가 자연스러운 접근.

## Action

**한 달씩 거꾸로 시간여행** 모델로 핵심 UX 재설계. 기존 Phase 5·6 화면은
검증 후 대체할 때까지 유지 (회귀 위험 최소화).

### 핵심 결정

| 항목 | 결정 |
|---|---|
| 진행 방향 | 현재 (2026.5) 부터 한 달씩 거꾸로 |
| 화면 구성 | "○년 ○월엔 이런 일이 있었어요" + 4섹션 사건 + 음악 |
| 사건 섹션 | 정치·사회 / 문화 / 스포츠 / 유행 (각 1~5개) |
| 사건 상호작용 | 기억나는 것만 체크/삭제 + 옆에 내 이야기 입력 |
| 입력 방식 | 텍스트 + 음성 녹음 → AI 다듬기 (RAG 가드) |
| 유행 처리 | 항목마다 노출 기간 (시작월~끝월) → 해당 기간 매월 노출 |
| 국내 음악 | 멜론 월간 1~10위 (수작업 시드) |
| 해외 음악 | 빌보드 "그 무렵 대표곡" + 노출 기간 |
| 음악 재생 | 유튜브 검색 링크 (임베드 금지) |
| 음악 카드 | 이미지 없이 순위 타이포 + 시대의 색 + 재생 아이콘 |
| 검증 범위 | 2025.6~2026.5 (12개월) 먼저 |

### 6단계로 분할 (Phase T1~T6)

순차 진행으로 각 phase 가 다음 phase 의 입력이 되게:

1. **T1 데이터 모델** — `MonthEvent` + `ChartSong` + 4 enum. 기간 노출 규칙
   `start*12+startM ≤ targetY*12+targetM ≤ end*12+endM` 한 SQL 로 표현.
2. **T2 시드** — 12개월치 사용자 큐레이션 데이터 적재 (사건 46건 + 음악 128건).
3. **T3 월 화면** — `/timemachine/[year]/[month]` 라우트. 4섹션 사건 + 텍스트
   음악. 양방향 내비 (← 이전 달 / 다음 달 →). EventItem 3-state 토글.
4. **T4 음성 → AI 다듬기** — Web Speech API 받아쓰기 + Claude RAG 가드 다듬기 +
   토큰 차감 (`chargeOneShot`).
5. **T5 음악 카드** — 이미지 0 / 저작권 0 / `eraColor` → palette 매핑.
6. **T6 UserMemory 통합** — 한 달 = N+1 행 (남긴 사건당 1행 + 월 회고 1행). 가족
   공유 자동 연결.

### 비정규화 선택 — `UserMemory.title = MonthEvent.title` 사본

타임머신 사건 추억은 `monthEventId` FK + title 사본 둘 다 저장. 이유:

- FK 만 두면 `MonthEvent` 시드 재실행 시 그 사건 title 변경이 추억까지 바뀜
- title 사본만 두면 "어떤 사건이었나" 추적이 끊김

→ 둘 다 두고 `onDelete: SetNull` — 시드 재실행으로 MonthEvent 가 갈려도
추억은 살아남고 title 사본 보존. monthEventId 만 null 이 됨 (UI 가 "옛 사건"
구분 안 함은 H1 픽스로 별도 해결: deterministic id 로 시드 재실행 시 id
유지).

## Result

한 세션에 T1 → T6 + 자체 검토 16건 + 시급 6건 픽스까지 마침.

### 검증

- 12개월 (2025.6~2026.5) 풀 데이터 ✓
- 사건 4섹션 + 국내 10곡 + 해외 3~6곡 매월 노출
- 기간 항목 (유행 / 해외음악) 이 여러 달 걸쳐 정상 노출 (8월 화면에 6~8월 한정
  '키캉스' / 8월만 한정 '광복 80주년' 둘 다 잡힘)
- 라운드트립: 사건 "남기기" + 메모 + 월 회고 → UserMemory 4행 (사건 2 + 회고
  1 + Phase 7 행 1 보존) → 새로고침 시 복원 ✓
- 가족 공유: 룸 멤버가 룸 주인의 타임머신 사건/월 회고/Phase 7 추억을 한 화면에
  봄 ✓ (`lib/rooms.ts` 무수정)

### 트레이드오프

- **기존 Phase 5·6 화면 유지** — 검증 전엔 안 지움. `/timeline` 헤더 진입은
  살아있고 `/timemachine` 가 새 주력 진입. 운영 안정성 우선.
- **시드 12개월만** — 그 너머 확장은 검증 후. 사용자가 직접 큐레이션해야 해서
  대규모 자동 시드는 못 함 (저작권·정확성).
- **타임머신 음악 = 카탈로그 (사용자 추억 미연결)** — ChartSong 은 단순 노출용.
  음악 트리거에 사용자 추억을 매기는 흐름은 후속 phase 에서.

### 일반화된 학습

1. **데이터 모델 → 시드 → 화면 순서가 정답** — T1 (모델) 을 가장 먼저 잡으니
   T2 (시드) 부터 검증 가능, T3 (화면) 은 데이터 준비된 상태에서 디자인만
   집중. 중간에 모델 흔들리는 일 없었음.
2. **검증 범위를 일부러 좁히기** — 12개월만 먼저 만들고 검증. 50년치 시드 +
   화면을 동시에 만들었으면 어느 한쪽이 깨질 때 원인 격리 어려움.
3. **기존 테이블 재사용 (UserMemory)** — T6 에서 `TimemachineMonth` 별도 모델
   대신 `UserMemory` + `createdVia` 디스크리미네이터로. 가족 공유 / 책 제작
   파이프라인이 자동으로 새 데이터를 흡수.
