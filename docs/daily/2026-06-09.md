### Auth.js 디버깅 — 두 가지 문제 동시

**증상:** `getUserByAccount` 가 `Authentication failed against the database server, the provided database credentials for `postgres` are not valid`. session 200 + null 은 정상이라 JWT decode 만은 통과(DB 조회 0). 실제 DB 쿼리에서만 실패.

**의심 1 → 기각:** dev 서버가 옛 globalThis Prisma 캐시 → 서버 죽이고 재시작했는데도 같은 에러.

**확정:** `awk` 로 `.env` 의 라인 형태(길이/따옴표) 분석 → **DATABASE_URL 이 두 줄 활성**. Line 1 에 옛 로컬(`postgresql://app:app@localhost:5432/lifelog`) + Line 35 에 Supabase. ## 3. 사진 기능 — 2단계 (Photo 모델 + 정식)

### 데이터 모델

옵션 C (Photo + UserMemory 1:N):
- 새 마이그 `20260609000000_add_photo` — Photo 테이블 10 컬럼 + 2 인덱스 + 2 FK(User cascade, UserMemory cascade)
- UserMemory 에 `photos Photo[]` 역참조 한 줄
- 디스크리미네이터 일관: 모든 사진 = UserMemory 1행 (createdVia="photo") + Photo 1+ 장
- 인물/장소/year/month/title 미러링으로 룸·반응·진척 자동 호환

### 헬퍼 — `lib/photos.ts`

- `createIndependentPhoto(userId, input)` — Storage 업로드 먼저(path 확정) → DB transaction(UserMemory + Photo) → DB 실패 시 try/catch 로 Storage 롤백 (orphan 방지 핵심)
- `deletePhotoOwned(userId, photoId)` — 권한 검증 → Storage remove 먼저 → DB transaction(Photo delete + photo-only 메모리도 정리). life_event 첨부 사진(3단계+)이면 메모리 보존
- `listUserPhotos(userId)` — DB findMany + signed URL 병렬 prefetch, limit 200

### API + UI

- `POST /api/photos` — multipart + 검증 6단계(auth, 용량, mimeType, HEIC, magic number, year/month/caption) + createIndependentPhoto
- `DELETE /api/photos/[photoId]` — 권한 + deletePhotoOwned
- `/photos` — 그리드 + 업로드 폼(year/month/caption 입력) + 클릭 시 풀스크린 모달(닫기/삭제) + 옵티미스틱 hide

### 정합성 — orphan 방지 흐름

```
업로드: Storage put → DB tx (UserMemory + Photo)
         └ DB 실패 → try-catch 가 Storage 롤백

삭제:   Storage remove → DB tx (Photo delete + photo-only 메모리 정리)
         └ Storage 실패 → throw, DB 무영향
```

# 2026-06-09 (세션 2) — 사진 기능 3·4·5단계 + periodAnchor + 기간 자유추가

## 한 줄 요약

어르신이 올린 사진을 **인생 연혁에 표시**(3)하고 **이벤트에 첨부**(4)하는 풀스택 완성. 기간
이벤트(학교/군대/직장)의 시작·끝 두 점에 사진을 따로 배치하는 **periodAnchor** 추가. 자유
추가 폼에서도 **기간(시작~끝) 입력** 가능하게. 마무리에서 **독립 사진이 가족 룸에 새던 leak
1건 발견·픽스**.

## 1. 사진 3단계 — 인생 연혁 표시

- `getLifeEvents` **순수 DB 유지**가 핵심 — signed URL(Storage I/O)을 넣으면 test 스크립트가
  자격증명 없이 못 돌린다. 그래서 `getLifeEvents` 는 photos 경로만 들고 오고, **page.tsx(RSC)
  가 signed URL 배치 발급**(`Promise.all` + 개별 try/catch → 실패분만 누락).
- `LifeEvent.kind` 3종(`life_event`/`era_event`/`photo`) + `photos[]`. `createIndependentPhoto`
  에 `eventYear`/`eventMonth` **미러링**(이거 없으면 `where: { eventYear: not null }` 에서 사진이
  통째로 빠짐). 기존 행 백필 스크립트(no-op — 2단계 테스트 사진은 이미 삭제됨).
- 3종 시각: life=amber / era=slate / **photo=sky+📷**. 독립 사진 행(sky 카드, 썸네일이 주인공)
  + life_event 첨부 strip(amber 카드 아래 최대 3장+"외 N장"). 라이트박스 **보기 전용**.

## 2. 사진 4단계 — 편집 화면 첨부

- `attachPhotoToMemory(userId, memoryId, input)` — **검증을 Storage 업로드 전에**(소유+life_event)
  → 거부될 요청이 orphan 파일 안 남김. POST `/api/photos` 에 `memoryId` 분기.
- `EventPhotos` 형제 컴포넌트(EventForm 무수정). 첨부/삭제. 취소·저장 버튼은 EventForm 의
  `children` 으로 사진 섹션을 받아 **항상 맨 아래**.

## 3. periodAnchor — 기간 이벤트 시작/끝 사진 분리

- **문제**: 기간 이벤트는 `expandPeriods` 가 시작·끝 두 점으로 split 하는데, `makeEndRow` 가
  `...p.src` 로 photos 까지 복제 → 입학 사진·졸업 사진이 양쪽에 다 떴다.
- **해결**: `Photo.periodAnchor String @default("both")` (마이그 `20260609010000`, ADD COLUMN 한
  줄, 기존 행 자동 both). `start/end/both` + PATCH `/api/photos/[id]` 재태그. `expandPeriods` 에
  `isPeriodStart` 마킹 + `PhotoStrip` 앵커 필터(끝=end/both, 시작=start/both, 단일=전부).
- 라벨: 처음엔 카테고리별(입학/졸업·입대/제대·입사/퇴사) → 사용자 피드백("입학/졸업은 학교에만")
  으로 **공통 `시작 무렵 / 기간 전체 / 끝 무렵`** 통일. `category` prop 제거(고아).

## 4. 자유 추가 폼에서 기간 입력

## 5. 마무리 — 룸 leak 픽스

- **발견**: `listRoomMemories` 가 createdVia 필터 없이 멤버의 모든 UserMemory 노출 → 독립 사진
  메모리(`createdVia="photo"`, year/title 미러링)가 **가족 룸에 이미지 없는 텍스트 카드로 샘**.

- **픽스**: `where` 에 `createdVia: { not: "photo" }`. era_event 는 E2/E3 정책상 노출 유지.
  회귀 테스트 `test-photo-room-isolation.ts` 신규.
