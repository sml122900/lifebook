## A) 새 기능 — 인물 + 장소

### A1. 인물 P1 — 데이터 모델 + 헬퍼

**스키마 (마이그 `20260603113549_p1_person_and_person_event`)**:

```
Person                          PersonEvent
─────                           ───────────
id (cuid)                       id (cuid)
userId ◀── User.cascade         personId ◀── Person.cascade
name        max50               memoryId ◀── UserMemory.cascade
relation?   max30               userId      (권한 비정규화)
metYear?    1900~now+1          createdAt
memo?       max100              @@unique([personId, memoryId])
@@index([userId])               @@index([userId])
                                @@index([memoryId])
```

`UserMemory.personEvents PersonEvent[]` 역참조 → UserMemory 삭제 시 PersonEvent cascade. `Person/PersonEvent` 는 추가만 — 룸·반응·진척·기존 createdVia 어디도 영향 X.

**헬퍼 9개 (`lib/people.ts`)**: CRUD 5개 + linkPersonToEvent/unlinkPersonFromEvent + listPeopleByEvent/listEventsByPerson. 전부 `userId` 첫 인자, `findFirst/updateMany/deleteMany {id, userId}` 패턴으로 권한 검증.

- `linkPersonToEvent` 가 `LinkResult = "linked" | "already" | "not_found" | "not_life_event"` 4종 enum 반환 — `createdVia="life_event"` 만 허용 (인물은 *인생* 이벤트에만, timemachine_event·ai_chat 거부).
- P2002 catch 로 idempotent 토글 (동시 클릭 race-safe).

**검증 (`db/test-people.ts`)**: 39 assertion — CRUD/권한 14, link/unlink/idempotent/life_event 가드 12, cascade 4, 정렬/권한 9. 모두 통과 + 기존 검증 (life-events, life-event-crud, room-create, attendance, charge, family-reactions[기존 markSeen 2건 외]) 회귀 0.

### A4. 장소 매칭 — 데이터/API/UI

**스키마 (마이그 `20260603134442_place_fields`)**: `UserMemory` 에 5 컬럼 추가 (모두 nullable)
```
placeName    String?   -- "강원도 춘천시 근화동"

lng          Float?
```

기존 행 모두 null — 무영향. life_event 미러링 패턴 유지.

**`/api/place-search` 라우트** (POST `{query, source}`):

- query 1~100 자 trim

- 구글: `places.googleapis.com/v1/places:searchText` + `X-Goog-Api-Key` + `X-Goog-FieldMask` (필드 마스크 필수)
- 5초 타임아웃 (AbortController 적용은 픽스 패스에서 — M12 참조)

처음엔 "auto" 분기 (한글 정규식 → 네이버) 였으나 사용자 의도와 어긋날 수 있어 (한글로 "Tokyo Tower" 검색) 사용자가 직접 골라야 하는 UI 로 변경: 큰 버튼 2개 (🗺️ 네이버 / 🌍 구글) → 텍스트 입력 + 결과 → 선택.

**`PlaceSearchInput.tsx`** — 3단 화면:

- B) source 선택 후: 검색 입력 + 결과 5개 + 지도 타일
- C) 선택 완료: 📍 카드 + 작은 미리보기 지도 + [다른 곳으로 바꾸기] [지우기]

## B) v3 진단 후 11항목 픽스

기능 작업 끝낸 뒤 신규 코드(lib/life-events·lib/people·lib/age·lib/user-entry·lib/side-panel-data·lib/life-record/questions·app/life-record·app/life-timeline·app/people·app/enter·app/components/PlaceSearchInput·app/components/maps·app/api/place-search·prisma/schema.prisma 신규 영역) 한 패스 진단. 60+ 항목 발견 → 심각도(높음/중간/낮음) 분류 후 보고. 사용자 결정으로 H 7개 + M 4개만 즉시 픽스, 나머지는 후속 묶음으로 연기.

### B1. H1 — safeReturnTo open redirect 차단

`app/people/new/page.tsx` 의 `safeReturnTo` 가 `startsWith("/") && !startsWith("//")` 만 검사 → `/\evil.com` 백슬래시 우회 통과. 브라우저가 백슬래시를 슬래시로 정규화하면 외부 호스트로.

`URL` 객체로 정규화한 뒤 origin === dummy + 재구성 문자열 === 원본 둘 다 검증. 백슬래시·다중 슬래시·encoded 우회 모두 차단.

### B2. H2 — skippedLifeCategories race condition

`lib/life-events.ts` 의 `markCategorySkipped` / `unmarkCategorySkipped` 가 `findUnique → check → update` 3단계, 트랜잭션 밖. 동시 두 카테고리 건너뛰기 또는 "건너뛰기 + 답 저장" 충돌이면 후행 쓰기가 선행을 덮어써 데이터 손실.

`prisma.$executeRaw` + Postgres `array_append(array_remove(..., cat), cat)` 단일 statement. read-modify-write 사이 race window 제거. enum cast (`::"LifeCategory"`) 로 placeholder 처리.

### B3. H4·H5 — RenderEvent.originalId + expandPeriods 정렬 제거

H4: `TimelineView` 의 `e.id.slice(0, -4)` 매직 (`":end"` 4글자 가정) 으로 endYear split 행의 원본 lookup. id 형식 변경 시 silently 잘못된 origId 가 PeopleConnectModal 로 흘러가 다른 이벤트의 인물을 토글할 수 있음.

- `RenderEvent` 에 `originalId: string` 명시 필드 추가
- `expandPeriods` 가 split 행에 `originalId: e.id` 직접 세팅
- `computePeriodFlags`, 모달 호출, peopleByEvent lookup 모두 `e.originalId` 사용
- 매직 suffix 도 `":end"` → `"--end"` 로 분리 (id 충돌 가능성 더 낮춤)
- `origMemoryId()` 함수 제거

H5: 같은 함수에서 `sort by timeKey` 가 DB 정렬(`year ASC, month NULLS LAST, createdAt ASC`)과 미세하게 불일치. 같은 해 정확월(앵커) 와 사이이벤트(month=null) 의 timeKey 점수가 가까워 `Array.sort` 안정성에 의존.

`expandPeriods` 가 DB 순서를 그대로 보존하고, endYear split 시 시작 행 위치 기준으로 끝 행을 적절한 위치에 끼우는 `flushPendingBefore` 패턴으로 재작성. createdAt 정보까지 보존.

### B4. H6 — computeYearRange 빈 배열 가드

`Math.min(...[]) === +Infinity` / `Math.max(...[]) === -Infinity`. LineClickArea 가 NaN year 로 push. 현재는 `page.tsx` 의 `hasEvents` 가드로 보호되지만 방어 없음. `events.length === 0` 시 현재 연도 기본값 (`{min: now, max: now}`) 반환.

### B5. H7 — placeSource 정규화 통일

`validatePlace` 두 곳(life-record/actions·life-timeline/actions)에서 placeSource 가 화이트리스트 아니면 source 만 null 로 떨어지고 lat/lng/placeName 은 저장됨 → 외부 지도 링크 폴백이 어중간하게 다른 사이트로.

placeSource 가 "naver"/"google" 둘 다 아니면 전체 데이터 null 화 (전부 저장 안 함). 데이터/UI 일관성 회복.

### B6. M3 — 조사(과/와, 을/를) 자동 처리

`{person.name}과 함께한` 같은 고정 조사 → "영희과" 같은 받침 불일치.

`lib/josa.ts` 신규 — `hasBatchim` + `withJosa(name, "과/와"|"을/를"|"이/가"|...)` + 단축 `objectJosa`/`subjectJosa`. FamilyNewsCard 의 inline `objectJosa` 를 이 lib 으로 통합 + `people/[personId]/link/page.tsx` 의 "함께한 사건" 헤더에 `withJosa(name, "과/와")` 적용.

### B7. M7 — startEditing query prefill

`PlaceSearchInput` 의 "다른 곳으로 바꾸기" 누르면 입력창이 비어있음. `setQuery(value.placeName ?? "")` 추가 — 기존 장소명 prefill 로 작은 수정 후 재검색 자연스러움.

### B8. M9 — Esc IME 가드

`AssistantModal`/`PeopleConnectModal` 의 전역 Esc 핸들러가 한국어 IME 조합 중 Esc 도 모달 닫음 → 조합 취소 의도가 입력 손실로.

`e.isComposing` + `target.tagName === "INPUT"/"TEXTAREA"` + `isContentEditable` 가드 추가. IME 조합 시·입력 위젯 안 Esc 는 무시.

### B9. M12 — AbortController

`/api/place-search` 의 `withTimeout` 이 `Promise.race` 만 — timeout 후에도 underlying fetch 가 계속 돌아 소켓/메모리 누수.

`fetchWithTimeout` 신규: `AbortController` + `setTimeout(() => ctrl.abort(), ms)` 패턴. signal 로 진짜 cancel. AbortError 는 한국어 메시지로 정규화.

### B10. M20 — 구글 placeholder 한국어

`"Search a place… (e.g., Tokyo Station)"` → `"🔍 장소 검색… (예: 도쿄역, Eiffel Tower)"`. 시니어 친화 톤 일관.

## 결정 / 트레이드오프

- **인물은 *인생 이벤트* 에만** — timemachine_event/ai_chat 같은 다른 createdVia 메모리에는 인물 연결 금지. P1 헬퍼에서 `not_life_event` enum 으로 명시. 시각화·룸·반응 전부 life_event 단위만 다룬다는 v3 정책 일관.
- **장소 자동 분기 제거** — 처음 도입했던 한글 정규식 기반 auto 분기는 의도와 어긋나는 케이스(`Tokyo Tower` 를 한글로) 발생. 사용자가 명시적으로 네이버/구글 큰 버튼으로 선택.
- **모든 카테고리에 장소** — 처음엔 8 카테고리 게이트. 사용자 결정으로 결혼식장(RELATIONSHIP)·자녀 태어난 곳(FAMILY) 도 의미 있어 게이트 제거.
- **`placeSource` 미지원 시 전체 null** — 어중간한 lat/lng 만 저장 vs 전부 거부의 트레이드오프 중 후자. 외부 지도 링크 분기 일관성 우선.
- **expandPeriods 정렬 제거, DB 순서 보존** — TimelineView 가 자체 timeKey 정렬하던 것이 미세하게 DB 순서와 어긋남. 단일 진실 원천(DB) 우선.
- **Postgres native array + raw SQL** — `skippedLifeCategories` race 픽스. Prisma JSON·일반 array update 가 트랜잭션 단위 보장 어려워 raw 가 가장 간결.
- **AbortController** — 5초 timeout 떨어진 fetch 가 계속 돌면 누수. signal 끊기로 네트워크 레벨 정리.
- **lib/place-types.ts 분리** — `lib/life-events.ts` 는 prisma(node-only pg) 의존이라 클라 컴포넌트가 `EMPTY_PLACE` 값을 가져가면 dns 모듈 빌드 오류. 순수 타입/상수 파일로 분리해 클라/서버 공용. (오늘 작업 중 빌드 오류 → 즉시 픽스)
