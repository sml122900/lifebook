### Phase 8 — 토큰 결제 (MVP 완성)

8단계 + 사전 baseline 측정으로 정책 정확도 확보:

- **Baseline 측정**(`db/test-memory-usage.ts`) — 광화문 연가 / 강남스타일 / IMF 세 페르소나로
  추억 1건 평균 1,113 AI 토큰 (variance < 2%) 확인. 이 수치 위에서:
- **8.1 정책 상수**: `N=2000`(1 서비스토큰 ≈ 1 사이클), 신규 30토큰, 1,000원=100토큰(원가 1.9원
  → 마진 ~5배). `lib/tokens/policy.ts` 한 파일에 모음.
- **8.2 지갑 + 무료 지급**: `TokenWallet`(@unique userId) + `TokenTransaction`. 멱등 3중 가드
  — fast path / DB unique / P2002 catch. `events.createUser` + `jwt callback` backstop으로
  기존 사용자도 다음 로그인에 자동 grant.
- **8.3 사이클 단위 차감**: `AIMessage.chargedAt` 컬럼으로 unsettled 표시. settle 시 guided + summary
  토큰을 합쳐 한 번 `ceil` → 호출당 별도 ceil로 2배 부과 문제 해결.
- **8.4 잔액 부족 게이트**: `MIN_BALANCE_TO_START_CYCLE=2`. 페이지 진입(신규 cycle) + 답변 제출
  이중 가드. 시니어 친화 안내 카드(`InsufficientBalanceCard`)로 부담 없이 충전 유도.
- **8.5 토스 결제 (테스트 모드)**: `TokenOrder` 모델 + paymentKey @unique. 클라이언트는
  packageId만 전송, 서버가 policy에서 krw 결정. successUrl 도착 후 토스
  `/v1/payments/confirm` 호출 → `confirmed.totalAmount === order.krw` 매칭 시에만 적립.
  4중 검증(토스 측 + 서버 amount + userId scope + paymentKey unique).
- **8.6 잔액·내역 화면**: `/billing` 한 페이지에 잔액·패키지·거래 50건. reason별 한글 라벨,
  ±델타 tabular nums.

### Phase 9 — 가족 공유 (트랙 A + B)

권한·프라이버시 복잡도가 가장 큰 phase. 트랙 분리로 안정적 진행:

- **9.1 룸 모델**: `SharedRoom` + `RoomMember`(consentAt nullable). owner는 생성 즉시 consentAt 자동,
  invitee는 9.2에서.
- **9.2 초대 + 동의 게이트**: `RoomInvite.token` = `randomBytes(32).toString("base64url")`
  256bit. `/invite/[token]` 페이지는 동의 화면일 뿐, **링크만으로 자동 합류 안 됨**. 클라이언트
  체크박스 + 서버 액션 재검증 이중 가드.
- **9.3 공유 타임라인**: `listRoomMemories`가 자체 멤버십 재검증 + `userId IN (consented 멤버)`
  로만 fetch. 룸 A 멤버는 그 룸 멤버의 UserMemory만 읽기 전용 표시.
- **9.4 댓글**: `Comment(roomId, targetType, targetId, authorId)` polymorphic. createComment에
  target memory가 같은 룸 소속인지 추가 검증. `deleteMany WHERE id+authorId`로 비작성자 차단.
- **9.5 SharedMemory 모델**: roomId 소유. createdById + lastEditedById로 작성자/편집자 추적.
- **9.6 공동 작성/편집**: 작성·편집은 모든 consented 멤버, **삭제는 작성자 OR 룸 owner**. 매트릭스
  검증(9가지) 통과.
- **9.7 통합 타임라인**: anchor + 개인 추억 + 공동 추억을 한 year 그룹 안에. 4색 절제
  (sky / amber / emerald / violet) + `TimelineLegend`로 의미를 색에 의존시키지 않음. **추억이 있는
  연도만** 렌더 → anchor 폭주 방지(시니어 빽빽 우려 해소).

## 결정

- **N=2000(관대 옵션)** — baseline 측정 평균 1,113 AI 토큰 위에서 cycle당 정확히 1 토큰. 무료 30토큰
  = 30 사이클 체험. 출시 후 회상 절정 외 곡 추천이 많아지면 N 재조정 여지.
- **음악 시드 = 하이브리드(MusicBrainz + 큐레이션)** — MB 매칭률 77%, 한국 곡 한글 검색 다수 실패.
  **시드 연도가 정본**, MB는 시드보다 빠를 때만 채택(재발매가 first-release-date에 잡히는 케이스 대비).
- **룸 = 명시 동의 게이트** — 초대 링크 클릭만으로 자동 합류 X. 256bit 토큰 + 클라/서버 이중 동의 검증.
- **error boundary 정책** — raw error.message는 절대 UI 비공개 (Prisma SQL, 외부 API echo 누설 방지).
  digest만 사용자에게 노출 (support 코드).
- **charge race fix는 schema 변경 없이 SQL로** — PostgreSQL의 row-level lock + WHERE 재평가가
  핵심. updateMany WHERE + RETURNING이면 두 동시 transaction 중 한쪽만 row 확보.

## 발견 / 트러블

- **InsufficientBalanceError가 "use server" export 위반** — "Only async functions are allowed to be
  exported" 빌드 에러. 클래스를 `lib/tokens/errors.ts`(use server 아님)로 분리하고 actions.ts는
  import만 (어제 commit `ecca72a`).
- **Voyage MusicBrainz 매칭률 한계** — 한국 곡(특히 K-pop) 한글 검색 실패 다수. MB의 first-release-date가
  재발매·리메이크를 가리키는 경우도 많음 (Imagine→1997, 광화문 연가→2002). 정책으로 "시드 정본, MB는 더
  빠를 때만 채택" 채택해 우회.
- **Schema와 코드 의도 불일치 1건** — `UserMemory.visibility` 컬럼이 있지만 어디서도 사용 안 됨.
  검토 3/6에서 발견. 컬럼 활용 또는 제거 결정은 후속 phase로.
- **`SharedRoom.ownerId: Cascade`** — owner 회원 탈퇴 시 가족 룸 전체 + 다른 멤버 작성 데이터까지
  cascade 삭제됨. owner 양도(transfer ownership) 메커니즘 필요 — 회원 탈퇴 기능 추가 전 정책 결정 필요.
- **AIMessage chargedAt race** — 이론적 race를 검토에서 발견 → 더블 클릭/네트워크 retry로 실제 발생
  가능 시나리오 확인 → SQL conditional update로 해결 (바구니 1-1).
