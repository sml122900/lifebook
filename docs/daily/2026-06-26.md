## 1. 포스터 맞춤배경 "결정" 503 — 근본 픽스 (base64 왕복 제거)

### 배경

`/poster/custom` 에서 맞춤배경 만든 뒤 "이 배경으로 결정"이 **간헐 503**. 클라 가드로
화면 이동은 됐지만 서버 저장이 불확실했다.

### 진단 (DB SELECT 가 결정타)

운영 DB 읽기 전용 점검 — `Poster.customBgPath` 가 채워진 행이 **존재**(작은 그림은 저장
성공한 적 있음). 즉 503 은 *항상*이 아니라 **이미지가 클 때만** = base64 크기 문제 확정.

- 원인: "결정" 시 미리보기 PNG 를 **base64(~16MB)로 서버 액션 인자**에 실어 전송.
- `next.config` 의 `serverActions.bodySizeLimit: 16mb` 는 **Next 내부 파싱 한도만** 올린다.
  Vercel 서버리스 함수의 **요청 본문 플랫폼 한도(~4.5MB)** 는 이걸로 못 올림 → 큰 요청은
  함수 도달 전에 끊겨 503. (설정해도 안 막힌 이유.)

### 픽스 — 생성 즉시 Storage 업로드, 결정 시 경로만

- `generateCustomBackground`: 생성 직후 `uploadPosterBackground` 로 Storage 업로드 →
  **경로(`bgPath`) 반환**. 버퍼는 서버 생성물이라 요청으로 오간 적이 없음 → 한도 무관.
  base64 는 미리보기 표시용으로 *응답*에만(응답은 요청 본문 한도 대상 X). 다시 만들기 시
  직전 미리보기 정리(`prevPath`, 세트 내 orphan 방지).
- `saveCustomBackground(bgPath)`: base64 → **경로 문자열만** 수신(소유 경로 정규식
  `^poster-bg/{userId}/\d+\.png$` 가드). 요청 본문 수십 바이트 → **503 원천 소멸**.
- `next.config`: 불필요(+무효)해진 `bodySizeLimit 16mb` 제거.

트레이드오프: 생성 후 결정 안 하고 이탈하면 그 그림 1장이 Storage orphan(이전엔 0). 같은
세트 다시 만들기는 직전 것 정리로 누적 X. 기존 "Storage cleanup cron" 후속으로 처리.

커밋 `f47b9dc`(초기 try/catch·폴백) → `076ce96`(근본 픽스).

## 2. 토큰 잔액 사이드 패널 실시간 갱신

### 배경

토큰 구매·차감 후 우측/사이드 패널 잔액이 **새로고침 전까지 stale**.

### 원인

사이드 패널 잔액은 루트 레이아웃(`app/layout.tsx`)이 요청당 1회 `loadSidePanelData →
getBalance` 로 읽어 클라 `SidePanel` 에 **정적 prop**으로 박는 값. App Router의 **공유
레이아웃은 클라 네비게이션 때 재실행 안 됨** → mutation 이 레이아웃을 무효화하지 않으면
새로고침 전까지 stale. (`getBalance` 의 `cache()` 는 요청 내 중복 제거용, 문제 아님.)

### 픽스 — 차감·구매 지점에서 레이아웃 갱신

- 배경 생성 차감: `setBalance`(로컬 즉시) **+ `router.refresh()`**(패널 동기화).
- 토큰 구매: `SuccessScreen` 클라화 → 마운트 시 1회 `router.refresh()`. 단 새로고침하면
  서버 페이지가 idempotent 분기("이미 충전됨")로 다시 그려지므로, 표시값은 `useState`로
  **고정**해 메시지가 뒤집히지 않게.
- 출석 보너스: `revalidatePath("/timemachine")`(redirect-only라 무의미) → `("/", "layout")`.
- companion 유료 채팅·refine 다듬기: 차감 직후 `router.refresh()`.

발견된 5개 흐름(구매·배경·출석·companion·refine) 모두 커버. 커밋 `fb98a0c`.

## 4. 동반자 C1 — 시스템 프롬프트 v3 강화 (지속 검증·시간순·정리본)

### 배경

어르신은 연도 착오·순서 혼동이 잦다. 동반자가 소극적 기록만 하지 말고 **지속 검증·정리**
하도록 프롬프트 강화. 기존 "가장 중요한 원칙 — 자연스러움(질문 폭격 금지·한 번에 하나·
기억 반박 금지)"을 깨지 않게 타이밍 가드를 명시해 녹임.

`lib/companion.ts` `COMPANION_SYSTEM_PROMPT_V3` 에 3섹션 삽입:

1. **사건의 시간 순서 — 받을 때마다 의식**: 연도뿐 아니라 상대 순서("결혼 전/후?")
   확인 + 연도 불확실 시 상대적 앵커("독일 가실 때 아이가 있으셨어요?").
2. **모순·착오 부드럽게 살피기 — 지속 검증**: 모순 감지해도 바로 안 끊고 화제 전환·정리
   시점에 확인. 단정 금지·질문 형식, 자존심 배려("틀리셨어요" ✗ → "제가 헷갈려서요" ✓).
3. **주기적 정리본**(기존 [되짚기] 확장): 예닐곱 턴·화제 전환·"정리해줘" 시 "지금까지
   정리해볼게요 📋" + 시간순 목록 → "맞을까요?"로 오류 자연 발견. 빈도 가드.

세 축이 맞물림: ①이 순서를 받아두고 → ②가 모순을 포착해 → ③ 정리본에서 확인.

## 6. 동반자 C3 — 연도 감지 + 그 시절 아카이브 인라인 카드

대화 중 연도 언급 시 그 해 시대 사건·히트곡을 인라인 카드로 → 기억 트리거.

- 감지: **클라 regex** `(?<!\d)(19\d{2}|20\d{2})(?!\d)` (1900~2099, 긴 숫자 오매칭 방지)로
  AI 응답 + 사용자 발화에서 연도 추출(서버 불필요).
- 조회: 첫 연도 감지 시 **기존 `getEraCatalog`(G2) 1회 로드** → 클라가 연도별 로컬 필터.
- 중복 방지: `shownYearsRef` Set(대화당 연도별 1회, 데이터 없는 연도도 처리됨 표시).
- 표시: `EraArchiveCard.tsx`(신규) — 메시지 아래 접이식 "🕰️ {연도}년, 그 시절 이야기 보기".
  사건 최대 3 + 노래로 합계 5까지, "이 노래 기억나세요?" 문구. 비블로킹(응답 즉시, 카드는
  카탈로그 해석 후 매달림).
- 데이터 없는 연도(시대 시드 1980~2023 밖, 예: 1954) → 스냅샷 미생성 → 카드 X.

⚠️ 검증 시 카드를 실제로 보려면 1980~2023 연도(예: "2002년")로 테스트. "1954년"은 감지는
되나 데이터 범위 밖이라 카드 없음.

## 미완료 후속 / 짚어둘 점

- **정적(tsc·build)까지만** 검증. 실제 동작은 화면·대화로 확인 필요:
  - 503: 큰 그림으로 결정 → 503 사라졌나.
  - 토큰: 구매·배경·companion·refine 후 새로고침 없이 패널 갱신되나.
  - 삭제: 편집 화면 삭제 → 다이얼로그 → 연혁에서 사라짐 / 남의 id 차단.
  - C1: 모순 넣고 6~7턴 후 정리본 나오나, 부드럽게 확인하나.
  - C2/C3: 패널 시간순·모바일 토글 / "2002년" 언급 시 카드, 중복 안 뜸.
- 503 근본 원인(Vercel 4.5MB 요청 본문 한도 vs next.config bodySizeLimit)은 트러블슈팅
  문서(`docs/troubleshooting/`) 후보 — 재발 방지 가치 있음.
- C3 오탐 여지: "2000원"처럼 연도 아닌 4자리 → 그 연도 데이터 있으면 카드 가능(드묾). 단순
  regex 정책 한계로 수용, 필요 시 "년" 접미 요구로 강화.
- Storage orphan(생성 후 이탈분) cleanup cron — 기존 후속과 동일 영역.
- companion C1 모델은 Haiku — "예닐곱 턴"은 전달 history 길이로 어림. 화제 전환·명시 요청이
