# 2026-06-10 — 사진 × 인물·장소 매칭(A·B·C) + 화면 테스트 중 잡은 SSR 버그 2건

직전 세션에 구현해 둔 **사진 인물·장소 매칭(A·B·C)** 을 사용자가 화면 테스트하다 `/life-timeline` 이 깨지는 걸 발견 → 두 개의 **별개 SSR 버그**를 잡고 전체 커밋·푸시.

## 한 줄 요약

- **A** 중복 버그 — 기간 이벤트(시작·끝 split)에서 인물·장소가 양쪽에 다 뜨던 걸 `isPeriodEnd` 점에서 suppress(앵커 없이, 시작 점에만)
- **B** 인물 매칭 — 사진 메모리(`createdVia="photo"`)도 인물 연결 허용. `LinkResult.not_life_event` → `not_linkable`, `LINKABLE_CREATED_VIA = {life_event, photo}` Set 가드
- **C** 장소 매칭 — 사진 업로드/카드에 `PlaceSearchInput` 재사용. `validatePlace` 를 `"use server"` 밖 순수 모듈로 추출, `updatePhotoMemoryPlace` + `updatePhotoPlaceAction`
- **버그①** `"use server"` 파일이 number export — `app/era/actions.ts` 의 `ERA_MEMORY_LIMIT = 500` → 순수 모듈 `lib/era-constants.ts` 로 분리
- **버그②** `window is not defined` (콜드 SSR) — `@googlemaps/js-api-loader` 가 모듈 최상단에서 window 참조 → `PlaceMap` 을 `next/dynamic({ ssr:false })` 로

## 1. A·B·C — 사진에 인물·장소 얹기

직전 세션의 사진 3·4·5단계(연혁 표시·이벤트 첨부·periodAnchor)에 이어, 처음 목표였던 **사진 + 인물·장소 매칭**. 설계 먼저(구현 X) → A→B→C 순서로 각 단계 verify+회귀 후 진행.

### 결정 (사용자 확정)

| # | 결정 | 비고 |
|---|---|---|
| 1 | 독립 사진 편집 = **카드 버튼**(새 화면 X) | PhotoCard 에 📍·👤 버튼 |
| 2 | 첨부 사진은 **부모 life_event 의 인물·장소 상속** | 사진 자체엔 안 매김(독립 사진만) |
| 3 | `not_life_event` → **`not_linkable`** 리네임 | life_event + photo 둘 다 연결 허용 |
| 4 | 중복 버그 = **"시작 점에만"** (앵커 없이 `isPeriodEnd` suppress) | 인물·장소는 기간 전체에 걸친 것이라 사진의 periodAnchor 와 다름 |

### A — 중복 suppress (`TimelineView.tsx`)

기간 이벤트는 `expandPeriods` 가 시작·끝 두 점(같은 `originalId`)으로 split. 사진은 `periodAnchor` 로 어느 점에 띄울지 정하지만, 인물·장소는 *그 시기 전체*에 걸친 정보 → 끝 점에서도 떠 "입학 때 만난 친구가 졸업 점에도" 보였다. 해결: 양쪽 카드 영역에서 `PlacePreview`·`PeoplePreview` 를 `{!e.isPeriodEnd && (...)}` 로 감싸 **시작 점에만** 그림. 사진(periodAnchor)과 달리 Photo 컬럼 추가·마이그 0 — 순수 렌더 억제.

### B — 인물 매칭 (`lib/people.ts`)

- `LINKABLE_CREATED_VIA: ReadonlySet<string> = new Set([CREATED_VIA_LIFE_EVENT, CREATED_VIA_PHOTO])`
- `LinkResult` 의 `"not_life_event"` → `"not_linkable"`. 가드 `if (!LINKABLE_CREATED_VIA.has(memory.createdVia)) return "not_linkable"` — era_event 만 거부.
- `listEventsByPerson` 필터·매핑이 photo kind 도 구분(`kind: m.createdVia === CREATED_VIA_PHOTO ? "photo" : "life_event"`).
- PhotoCard 에 👤 버튼 + `onOpenPeople` prop. `PeopleConnectModal`·`LinkToggleRow`·era-stash 주석까지 `not_linkable` 로 통일.
- ⚠️ `lib/photos.ts` 의 `attachPhotoToMemory` 가 가진 **별개** `not_life_event`(첨부 도메인)는 리네임 X — grep 으로 people 도메인만 0건 확인.

### C — 장소 매칭 (`lib/photos.ts`, `lib/place-validate.ts`, `app/photos/actions.ts`)

- `validatePlace` 를 `app/life-timeline/actions.ts`("use server") 에서 **순수 모듈 `lib/place-validate.ts`** 로 추출 — "use server" 는 비-async export 불가라 공유하려면 밖으로 빼야 함(H7 정책: 미지원 placeSource → 전체 null).
- `createIndependentPhoto` 에 `place?: PlaceInfo` + 메모리에 5컬럼 저장. `updatePhotoMemoryPlace(userId, memoryId, place)` = `updateMany {id, userId, createdVia:"photo"}` 가드(life_event 거부).
- `PhotosUploadForm` 에 PlaceSearchInput("어디서 찍었나요?"). POST `/api/photos` 가 FormData 의 place 5필드 파싱(`numFromForm`) → validatePlace.
- PhotoCard 가 **장소 모달 self-manage**(prop threading X — `e.place` 가 이미 이벤트 객체에 있음. 인물은 RSC fetch 라 threading). 옵티미스틱 `localPlace` state + `PlacePreview` 칩.

## 2. 화면 테스트에서 드러난 SSR 버그 2건

C 가 `PlaceSearchInput` 을 **`/life-timeline`(TimelineView) 에 처음** 끌어오면서, 그동안 다른 라우트에만 있어 안 보이던 잠재 버그 2건이 메인 페이지에서 터졌다.

### 버그① — `"use server"` 파일이 number 를 export

```
Error: A "use server" file can only export async functions, found number.
```

`app/era/actions.ts`(맨 위 `"use server"`)가 `export const ERA_MEMORY_LIMIT = ERA_MEMORY_MAX_LENGTH`(=500) 를 내보내고 있었다(E3 의 알려진 후속). 평소엔 통과하다, 장소 저장 액션(`updatePhotoPlaceAction`)이 같은 `/life-timeline` 액션 번들에 합류 → 번들 전체 **재검증** → number export 가 걸림.

- **막힌 곳**: 클라(`EraMemoryEditor`)는 이 숫자가 필요한데 `lib/era-stash`(prisma 의존, 서버 only)를 import 못 함 → 그동안 "use server" 액션 파일을 통로로 우회하고 있었다.
- **해결**: prisma 의존 없는 순수 모듈 **`lib/era-constants.ts`** 신설(= `place-types.ts` 패턴). 서버는 `lib/era-stash` 가 import+재노출(기존 `@/lib/era-stash` 호출자 호환), 클라는 직접 import. CLAUDE.md 의 "ERA_MEMORY 상수 두 곳 정의 통일" 후속도 동시 해결.

### 버그② — `window is not defined` (콜드 SSR 첫 요청 500)

```
ReferenceError: window is not defined
  at module evaluation (app/components/maps/GoogleMap.tsx:3:1)  // @googlemaps/js-api-loader
```

`@googlemaps/js-api-loader` 의 dist 가 **모듈 최상단**에서 `const trustedTypes = window.trustedTypes`(line 47) 를 평가 → 서버 모듈 평가 시 throw. 클라 컴포넌트라도 Next 는 SSR 위해 서버에서 모듈 평가. C 전엔 `/life-timeline` 이 이 체인을 안 끌어왔는데 이제 메인이 노출(`/add`·`/photos` 는 이미 들고 있어 동일 위험을 갖고 있었음).

- **증상 패턴**: 콜드 컴파일 직후 첫 요청만 500, 워밍 후 200 — 그래서 "가끔 깨짐"으로 보였다.
- **해결**: `PlaceSearchInput` 에서 `PlaceMap` 을 **`next/dynamic({ ssr: false })`** 로 로드. 지도는 순수 인터랙션(SEO 가치 X)이라 SSR 제외가 정답. `/add`·`/photos` 도 같이 단단해짐.
- **증거**: 픽스 후 로그에 `✓ Compiled` 직후 첫 `GET /life-timeline` 이 **200**(이전엔 매 콜드 컴파일 첫 요청 500).

## 4. 교훈

- **"use server" 파일은 number 도 못 내보낸다** — 클래스만이 아니다(use-server-class-export.md 재발). 클라가 서버 전용 모듈의 상수를 필요로 할 때 액션 파일을 통로로 쓰면 잠재 폭탄. 순수 상수 모듈이 단일 진실 원천.
- **잠재 버그는 "새 진입점"에서 터진다** — 두 버그 다 코드는 그대로였고, C 가 `PlaceSearchInput`/`updatePhotoPlaceAction` 을 메인 페이지·메인 액션 번들에 *처음 합류*시키며 표면화. 모듈 그래프에 새 노드를 더할 때 재검증이 도는 걸 염두.
- **브라우저 전용 라이브러리는 `ssr:false`** — 모듈 최상단 window 접근은 SSR 에서 무조건 터진다. 콜드-only 500 은 "가끔"이 아니라 "항상(워밍 전)" — 메인 동선에 들어오면 반드시 잡아야.
