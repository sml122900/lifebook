# 결정 — 비서 답변을 DB(검증) vs 웹 검색(보조) 으로 분리

## Problem

타임머신 v2 "AI 비서" 가 사용자의 회상 질문에 답해야 한다. 단일 출처로
가면 곤란한 두 축이 있음:

- **LLM 단독 답** → 환각. "1995년 5월에 △△ 사건이 있었어요" 가 사실이 아닐
  수 있음. 회상 서비스에서 틀린 시그널은 신뢰 붕괴.
- **검증 DB 단독 답** → 커버리지. `MonthEvent` 시드는 큰 사건만, 음악은
  차트만. 사용자가 묻는 "그 시절 인기 드라마" 같은 취향성 질문은 DB로 못 덮음.

또한 출처별로 **위험·비용·확신도** 가 다르다:

| | DB | 검색 |
|---|---|---|
| 정확성 | 수동 검증 ✓ | 출처 따라 다름 |
| 비용 | 0 (조회) | $0.01/회 + 요약 토큰 |
| 톤 | 단정 가능 | 조심스러워야 |
| 저작권 위험 | 0 | 가사·기사 인용시 ↑ |

한 통로(예: 항상 검색)에 몰아넣으면 단점이 곱해진다.

## Action

### 위험도 기반 3분기 라우팅

질문 키워드로 분류 후 DB 우선·검색 폴백:

| 카테고리 | 키워드 (예) | 1차 | 2차 |
|---|---|---|---|
| MUSIC | 노래·곡·가수·차트·OST | `ChartSong` 그달 노출 | 검색 |
| BIG | 사건·뉴스·정치·선거·올림픽·재난·IMF | `MonthEvent` POLITICS_SOCIETY+SPORTS | 검색 |
| TASTE | 드라마·영화·예능·유행·맛집 | (DB 미커버) | 검색 |

우선순위: MUSIC > BIG > TASTE. 매칭 실패 → TASTE로 폴백 (DB "없음"이라고
오답하는 것보다 검색이 안전).

### DB 답 = 템플릿 조립 (LLM 호출 0)

DB 결과가 있으면 LLM 거치지 않고 직접 한글 답 조립.

```
2025년 8월에는 이런 일이 있었어요.

• 한미 정상회담(워싱턴) — 이재명-트럼프 첫 회담, 관세·조선 협력 논의
• 한일 정상회담(도쿄) — 이재명-이시바 2차 회담
```

장점:
- **비용 0** — 사용자 토큰 차감 없음
- **결정적** — 같은 질문은 항상 같은 답
- **검증 가능** — 출력 텍스트가 DB 행과 직접 대응

음악도 동일. 국내 top5 + 해외 top3 까지만 (시니어 가독성 + 답 길이 제한).

### 검색 답 = Claude `web_search` 도구 + 가드 프롬프트

`lib/ai.ts` 에 `chatWithWebSearch()` 별도 함수. 기존 `chat()` 은 무수정.

```ts
const tools = [{
  type: "web_search_20250305",
  name: "web_search",
  max_uses: 1~2,
}];
```

웹 검색은 Anthropic 인프라가 실행하는 **서버 도구** — 우리는 한 번의
API 호출로 검색 + 요약 + 인용까지 받는다. 응답 content 는
`text` / `web_search_tool_result` 블록 혼합이므로 본문은 text 합치고
인용은 두 블록의 url/title 을 합집합으로 수집.

### 가드 시스템 프롬프트 (검색 전용)

```
- 가사·기사 그대로 인용 금지. 사실(제목·이름·시점·순위)만 우리 표현으로.
- 단정 금지. "...였던 것 같아요" / "확인해보세요" 톤.
- 사용자 기억 대신 만들기 금지 (RAG 가드 — 정보를 꺼내 실마리만).
- 음악 답은 곡명·아티스트·순위만. 이미지·임베드 ❌.
- 민감정보(건강·정치·종교)는 사용자가 묻지 않은 한 강조 ❌.
- 2~4문장. 시니어 친화 어조.
```

`temperature: 0.4` (`voice-cleanup` 의 0.3 보다 조금 높음 — 자연스러운
요약 여지). RAG 가드는 v1 `voice-cleanup` 패턴 그대로 — 도메인이 다른
질문에도 일관 적용.

### 토큰 정책 — `chargeOneShot.surcharge` 파라미터 추가

기존 정책: `tokensFromUsage(in, out) = ceil((in+out) / 2000)`.

검색 운영 비용 ($0.01/회) 은 in/out 토큰엔 안 잡힘. 정책 함수를
바꾸지 않고 호출부에서 가산:

```ts
// before (voice_cleanup 호출부 그대로 동작)
chargeOneShot(userId, in, out, "voice_cleanup")
// after (검색 답에만 +1 가산)
chargeOneShot(userId, in, out, "timemachine_assistant_web", undefined, 1)
```

`surcharge: number = 0` 기본값 → 기존 호출부 무영향. 가산 토큰이 있으면
in/out 이 0 이어도 차감 발생 (cost 계산 분기).

### API 라우트 (`/api/timemachine/assistant`)

- 401: 미인증
- 400: question 비었거나 year/month 잘못
- 402: 잔액 부족 (`{ error: "insufficient_balance" }`)
- 502: 외부 호출 실패 / web_search 비활성 등 upstream 에러

UI는 402만 "충전하기" 안내, 나머지는 짧은 안내 + retry 권유.

## Result

### 실측 (V1 검증 4케이스)

| 케이스 | source | 차감 | 출처 |
|---|---|---|---|
| BIG/DB ("이때 어떤 사건이?" + 2025/8) | db | 0 | — |
| MUSIC/DB ("유행한 노래?" + 2026/4) | db | 0 | — |
| TASTE/web ("인기 드라마?" + 2025/8) | web | 9토큰 | 8건 |
| BIG miss → web 폴백 ("큰 사건?" + 1995/5) | web | 9토큰 | 9건 |

- DB 답: **확실히 무료, 결정적**
- 검색 답: 평균 **9토큰** (기획서 잠정 2~3 보다 큼 — Claude 가 검색 결과를
  컨텍스트에 적재해 in≈14k 토큰). 정책 그대로, V2 실사용 보고 재조정.
- 환각 톤 가드 작동: 검색 답에 "였던 것 같아요" / "확인해보세요" 표현 다수.
- 인용 자동 포함: V2 UI 에 출처 6건까지 표시.

### 트레이드오프

- **키워드 라우팅 = 거친 분류**. "이때 노래 사건" 같이 두 키워드 겹치면
  MUSIC 우선. 실사용 보고 AI 분류기 도입 결정 (비용 + 지연 vs 정확성).
- **검색 9토큰 = 비싸 보임**. 사용자는 한 질문에 평균 9개. 일반적인
  타이밍(달마다 2~3개 검색) 으로 보면 한 달 회상에 ~20토큰 → 신규 30토큰
  무료분 안에서 1.5달 가능. 정책으로는 acceptable 범위.
- **DB 답 = 시드 의존**. 시드가 12개월(2025.6~2026.5) 만 — 그 외 달은 모두
  검색 폴백. 시드 확장은 별도 phase (수작업 큐레이션 필요).

### 일반화된 학습

1. **위험도가 다르면 출처도 분리**한다. 한 통로에 강제로 묶으면 양쪽
   장점을 다 잃는다 (단정 톤 vs 안전 톤은 양립 불가).
2. **검증 가능한 답을 무료로**. DB 답에 LLM 거치지 않으니 비용 0이고,
   같은 입력은 항상 같은 답. 회상 서비스에서 자신있는 톤을 보장.
3. **가드는 시스템 프롬프트 한 곳에 강제**. 도구가 늘어도(웹 fetch 등)
   프롬프트는 한 번만 손대면 된다.
4. **정책 함수는 안 건드리고 surcharge로 차이 흡수**. 새 비용 모델이
   생길 때마다 `tokensFromUsage` 를 흔들면 기존 호출부가 깨진다. 호출부
   파라미터로 표현하는 게 안전.
