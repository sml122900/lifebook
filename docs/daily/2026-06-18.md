## 1. 이메일+비밀번호 회원가입·로그인 (`767a73f`)

### 목표
할머니 테스트 계정 생성용 — SNS 비밀번호를 모르는 어르신도 이메일+비번으로 가입·로그인 가능하게.

### 제약
- 기존 Kakao / Naver / Google OAuth **완전 무변**
- JWT 세션 전략 이미 `strategy: "jwt"` → 전환 불필요
- 비밀번호 **bcrypt(cost=12)** — 평문 저장 절대 금지
- Credentials 는 Node 런타임 필요 → `auth.config.ts`(Edge) 아닌 `auth.ts`(Node) 에만 추가

### 작업

**스키마 + 마이그레이션**
- `prisma/schema.prisma`: `User.passwordHash String?` (nullable — OAuth 계정은 null 유지)
- `prisma/migrations/20260617000001_user_password_hash/migration.sql`: `ALTER TABLE "User" ADD COLUMN "passwordHash" TEXT;`
- `npx prisma migrate deploy` 사용자 직접 실행 (운영 DB 영향 최소화)
- `npx prisma generate` 후 tsc 재검증

**인증 로직 (`auth.ts`)**
- `providers: [...authConfig.providers, Credentials({...})]` 패턴 — 기존 3종 뒤에 추가
- `authorize()`: 이메일 조회 → `passwordHash null` 시 OAuth 전용 계정으로 거부 → `bcrypt.compare()` → 통과 시 user object 반환
- `bcryptjs` 순수 JS (native 바인딩 없음 — Vercel serverless 호환)

**가입 흐름 (`/signup`)**
- `app/signup/actions.ts`: 이메일 형식·비번 8자 이상 검증 → 중복 이메일 사전 차단 → `bcrypt.hash(password, 12)` → `prisma.user.create` → `ensureWalletWithSignupGrant` → `signIn("credentials", {..., redirectTo: "/enter"})` 자동 로그인
- `app/signup/SignupForm.tsx`: `useActionState` 클라이언트 폼 (이름·이메일·비번, 오류 인라인 표시)
- `app/signup/page.tsx`: 메타 "회원가입" + SignupForm 래핑
- `proxy.ts`: `PUBLIC_PATHS`에 `/signup` 추가 (미들웨어 차단 해제)

**로그인 화면 (`/login`)**
- `app/login/actions.ts`: `credentialsSignInAction` — `AuthError` 잡아 한국어 안내, `NEXT_REDIRECT` re-throw (Next.js 네비게이션 핸들)
- `app/login/LoginCredentialsForm.tsx`: `useActionState` 클라이언트 폼 (이메일·비번 + 오류 표시 + /signup 링크)
- `app/login/page.tsx`: OAuth 버튼 아래 `<hr>` "또는" 구분선 + `<LoginCredentialsForm />`

### 설계 결정
- `authorize()`가 `user.passwordHash === null`이면 즉시 `null` 반환 → OAuth 계정으로 이메일+비번 로그인 시도 자동 차단 (에러 코드 누출 없음)
- 가입 후 자동 로그인 = `signIn("credentials", ...)` re-throw NEXT_REDIRECT 패턴 — 별도 redirect 코드 불필요
- `AuthError` vs `NEXT_REDIRECT` catch 분리 — `instanceof AuthError` 만 잡고 나머지는 re-throw

### 결과
- tsc 클린, 12개 파일 +325줄
- 기존 OAuth 3종 완전 무변 (테스트 통과)
- 어르신 UI 규격: 입력·버튼 `min-h-[56px]`, 본문 18px 기본

## 3. React.cache() 중복 쿼리 제거 (`8f9683d`)

### 목표
데모 전 안전 최적화 — 함수 시그니처·반환값 불변, cross-request 영향 없음.

### 패턴
```ts
// 변경 전
export async function getLifeEvents(userId: string) { ... }

// 변경 후
async function _getLifeEvents(userId: string) { ... }   // 선언부만 수정
export const getLifeEvents = cache(_getLifeEvents);     // 마지막 줄 추가
```

### 적용 함수 5개

| 함수 | 위치 | 효과 |
|------|------|------|
| `getLifeEvents` | `lib/life-events.ts` | AssistantWidget + page 중복 → 1회 |
| `listAssistantAnswers` | `lib/timemachine-assistant-saved.ts` | AssistantWidget + page 중복 → 1회 |
| `getBalance` | `lib/tokens/wallet.ts` | 미래 중복 방지 |
| `getAttendanceStatus` | `lib/attendance.ts` | 미래 중복 방지 |
| `getFamilyNewsCount` | `lib/family-news.ts` | 사이드 패널 + 메인 중복 방지 |

### React.cache() 동작 원리
- **request 단위 격리** — 같은 요청 내 동일 인자면 캐시, 다른 요청은 항상 fresh
- 인자 매칭: primitive(`string/number`) 는 값 비교, object 는 참조 비교
- `getAttendanceStatus(userId, now=new Date())` — 두 번째 인자 생략 시 `(userId)` 만 캐시 키 (Date 기본값은 캐시 키에 포함 안 됨)

### 결과
- `/life-timeline` 1 요청당 DB 쿼리 순감 **−2건** (getLifeEvents 2→1, listAssistantAnswers 2→1)
- 5 파일 +20줄 — 함수 본문 0줄 수정
- tsc 클린, 기능 동일

### 4. 연혁 → /poster 진입 버튼 (`678c75d`)

#### 목표
`/poster` 로 가는 앱 내 진입점이 없어 URL 직접 접속만 가능했던 문제 해소.
데모·실사용 모두 연혁 화면에서 바로 포스터로 이어지는 동선 필요.

#### 작업
`app/life-timeline/page.tsx` 의 `hasEvents` 액션 섹션 버튼 행에 amber 버튼 추가:

```tsx
<Link
  href="/poster"
  className="inline-flex min-h-[56px] items-center justify-center rounded-md
             bg-amber-500 px-6 py-3 text-lg font-semibold text-white
             hover:bg-amber-600 focus:outline-none focus-visible:ring-4
             focus-visible:ring-brand focus-visible:ring-offset-2"
>
  이 연혁으로 포스터 만들기
</Link>
```

#### 설계 결정
- 기존 편집 버튼들(outline 스타일)과 같은 행 마지막 위치 — 추가·관리·보강 후 자연스럽게 시선이 가는 흐름
- `amber-500` 채움 + 흰 글씨 — 편집 버튼과 시각 구분, 동시에 앱의 amber(인생) 톤 일치
- `hasEvents` 조건 안 — 이벤트 0건 신규 사용자에겐 비노출(포스터에 넣을 내용 없음)
- 포스터 엔진·에디터·`/poster` 페이지 무수정, 마이그 0

#### 결과
- `app/life-timeline/page.tsx` +6줄
- tsc 클린
