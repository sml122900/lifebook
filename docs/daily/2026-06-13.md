# 2026-06-13 — 문장 다듬기 UX 개편 + 모델 선택·차등 차감 + manage 404 픽스

전날(6-12) 붙인 "문장 다듬기"를 부모님 테스트 직전에 실사용 관점으로 다듬은 날. 다듬기 동선을 자동 저장으로 통합하고, 메인 연혁 카드에 회상을 표시하고, 깨진 입력 결함을 잡고, 무료 단일 모델을 정밀도 3종 + 차등 차감으로 확장했다. 마지막으로 "그 시절 둘러보기"에서 담은 사건의 manage 수정 링크 404를 잡았다.

## 한 줄 요약

- **다듬기 자동 저장 UX** — `RefineSection`을 `EventForm` 안으로 옮겨 라이브 `content` 수신. [글 다듬기] = 현재 textarea 자동 저장 → 교정 → 전/후 카드. "저장 후 재진입" 동선 제거(미저장 draft 미반영 오발 해소). 빈 칸이면 `aria-disabled` + 안내 토스트
- **연혁 카드 회상 표시** — 메인 `EventCard`가 제목만 그려 다듬은 글이 안 보이던 버그. content 미리보기(`line-clamp-2`) 추가 + `applyRefinedAction` revalidate에 `/manage` 누락 동반 픽스
- **적용 후 2단 상시 표시** — "원래 글 보기" 토글 폐기, 원래 글/다듬은 글 두 카드 상시(전/후 동일 스타일)
- **깨진 입력 결함** — 자모깨짐(ㅁㄴㅇ)·오타뭉침이 NO_CHANGE로 빠지던 결함. 프롬프트에 맥락 추정 교정 + NO_CHANGE 조건 엄격화. 문두 군말(근데 그니까) 제거 조항도 추가
- **모델 선택 + 차등 차감** — tier 3종(빠르게/꼼꼼하게/가장 정밀 = Haiku/Sonnet/Opus). 실제 교정본 저장 시에만 1/3/5토큰 과금. 잔액 부족 → 402 친화 메시지
- **manage 404 픽스** — era·photo 행이 life_event 전용 편집 화면으로 링크돼 404. `e.kind` 분기로 각자 화면(/era·/photos) 안내

## 1. 다듬기 자동 저장 UX (커밋 62cc250)

### 문제

전날 만든 다듬기는 "수정 저장 → 편집 재진입 → 글 다듬기" 동선이었다. 사용자가 textarea에 회상을 쓰고 바로 [글 다듬기]를 누르면, 미저장 draft가 DB에 없어 교정이 빈/옛 content를 읽고 **"고칠 곳 없음"으로 오발**했다. 또 `RefineSection`이 `EventForm`의 `children`이라 라이브 `content` 상태에 접근할 수 없었다.

### 결정·구현

- `RefineSection`을 **`EventForm` 내부**(회상 textarea 바로 아래)로 이동. `EventForm`에 `refine?` prop 추가 → 편집 모드에서만 렌더, 라이브 `content`를 prop으로 전달
- [글 다듬기] = `saveMemoryContentAction`(content 전용 저장, refined 3필드 초기화 정책 미러) → refine API → 전/후 카드. **저장→재진입 동선 제거**
- 빈 칸이면 `aria-disabled` + `opacity-60`, 클릭 시 "더 떠오르는 게 있다면 항목에 글을 입력해주세요." 토스트(onClick 가드)
- 신규: `saveMemoryContent`(lib/life-events) + `saveMemoryContentAction`(actions)

### 연혁 카드 회상 표시 버그

content 렌더 경로 전수 추적 결과: **메인 `EventCard`는 제목·날짜만 그리고 `e.content`를 아예 표시하지 않았다.** 그래서 `getLifeEvents`의 displayRefined 스왑이 메인 화면에 *원래부터 안 보였다*. content가 보이는 reader 화면은 `/manage`뿐인데, `applyRefinedAction`의 `revalidatePath`에 `/manage`가 빠져 거기서도 stale.

→ `EventCard`에 회상 미리보기(`!isPeriodEnd && content` → `line-clamp-2 text-lg text-ink-soft`) 추가. `applyRefinedAction`·`discardRefinedAction`·`saveMemoryContentAction` 모두 `revalidatePath("/life-timeline/manage")` 추가.

## 2. 적용 후 2단 상시 표시 (커밋 4583b01)

displayRefined 재진입 시 "원래 글 보기" 토글이 한 번 더 누르게 하는 마찰이라, 원래 글(canvas)/다듬은 글(banner) 두 카드를 전/후와 동일 스타일로 **상시 표시**. 상태 문구로 "연혁엔 다듬은 글, 원문도 보관" 명시. 재다듬기 버튼 유지. `showOriginal` state 제거.

## 3. 깨진 입력 NO_CHANGE 결함 (커밋 4a2db7b)

증상: "어 아니 그거 맞나? 잘 ㅁㄴㅇ으음ㅇ. 종료가 ㄴㅇ 된근가?" → 고칠 곳 없음으로 통과.

프롬프트 보강 2건:
- 허용에 추가: "의미를 알아볼 수 없는 자모 깨짐(ㅁㄴㅇ, ㄴㅇ 등)·오타 뭉침은 앞뒤 맥락으로 추정 가능하면 교정, 추정 불가하면 그 부분만 삭제"
- NO_CHANGE 엄격화: "정말 고칠 게 없는 깨끗한 글에만. 군말·자모깨짐·오타가 하나라도 있으면 교정"
- 문두 군말("근데 그니까") 제거 조항도 함께(첫머리 군말)

실측: → "아니 그거 맞나? 잘 모르겠음. 종료가 안 된 건가?" (자모깨짐 맥락 추정 교정 확인). `test-refine-lv2`가 인자로 문장 받게 보강.

## 4. 모델 선택 + 차등 차감 (커밋 fba4798)

### 결정

전날 무료 단일(Haiku) 다듬기를 정밀도 3종으로 확장. 무료→유료 전환(Haiku도 1토큰)을 사용자가 확정. 상세: `docs/decisions/memory-refine-model-tiers.md`.

### 구현

- `refineMemorySpelling(userId, memoryId, tier="haiku")` — `TIER_TO_MODEL`(haiku→4-5 / sonnet→4-6 / opus→4-7). `chat(model)`. opus 4.7은 `supportsTemperature` 가드로 temperature 자동 제외(실측 확인)
- 차감 = `chargeOneShot` + surcharge 트릭(`tokensFromUsageForModel − tokensFromUsage`) → 총 cost가 정확히 1/3/5배. **저장 앞에 차감** → 잔액 부족이면 저장 없이 `InsufficientBalanceError` → API **402** "토큰이 부족해요. 충전 후 다시 시도해주세요"
- **교정본 저장될 때만 과금** — NO_CHANGE·길이가드 탈락은 0토큰(모델은 실행돼 Anthropic 비용은 발생하나 사용자 차감 0)
- UI: 칩 3종 "빠르게(1토큰)/꼼꼼하게(3토큰)/가장 정밀(5토큰)", 기본 빠르게. 선택칩 = banner+brand 보더(디자인 토큰 칩 스펙·필 금지), `min-h-[56px]`. refined 시 "N토큰 사용 (남은 토큰 M개)" 안내
- API: body `tier` 검증(미지 값 400)

### 실측 (차감 동작)

| tier | status | 차감 | 잔액 |
|------|--------|------|------|
| haiku | refined | 1토큰 | 100→99 |
| sonnet | refined | 3토큰 | 100→97 |
| opus | no_change | 0토큰 | 100→100 |

(opus가 33자 단편에서 no_change → 차감 0은 "저장 시에만 과금" 결정을 그대로 입증. temperature 가드도 opus 호출에서 에러 0으로 확인.)

## 5. manage 404 픽스 (커밋 d7c75d8)

### 증상·진단

`/life-timeline/manage`에서 직접 입력 이벤트 수정은 정상인데 "그 시절 둘러보기"에서 담은 사건 수정은 404. 상세: `docs/troubleshooting/manage-era-photo-edit-404.md`.

근본 원인: `getLifeEvents`는 life_event·era_event·**photo**를 모두 반환하는데 manage가 kind 구분 없이 전 행에 `/[id]/edit` 수정 링크를 달았고, `getLifeEventById`는 `createdVia: life_event` 필터라 era·photo id → null → `notFound()`. era·photo는 같은 `UserMemory` 테이블이라 `e.kind`로 구분 가능.

### 해결 (B안)

`e.kind` 분기(manage 1파일, 백엔드 0):
- life_event: 현행 수정 + DeleteButton
- era_event: "그 시절 둘러보기"(/era) + "빼기는 연혁에서" 보조 안내
- photo: "사진 화면 열기"(/photos) 안내
- 행 자체는 목록 유지(제목·날짜)

검증: 3종 행 분기 매핑 실측(life_event→수정+삭제 / era_event→/era 안내 / photo→/photos 안내).
