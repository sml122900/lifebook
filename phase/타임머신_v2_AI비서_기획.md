# Lifebook 타임머신 v2 — "AI 비서" 방식 (대화로 기억 되살리기)

> **작성 배경**: 실사용 통찰. 기존 v1(그달 사건·음악을 다 펼쳐 보여주고 고르게)은 **읽을 텍스트가 너무 많고, 관심 없는 정보면 흥미가 식는다.** 정보를 "차려주는" 방식의 한계.
> **새 방향**: 정보를 펼치지 않는다. **빈 기억 입력칸이 메인, 옆의 AI 비서에게 물어보며** 기억을 되살린다. 정보를 push(차림)에서 pull(필요할 때 꺼냄)로 전환.
> **상태**: v1(T3~T6) 완전 대체. 단, 기존 코드는 검증 전까지 지우지 않고 비활성. 새 화면으로 기본 진입.

---

## 1. 핵심 전환 — "차려주기"에서 "물어보기"로

| | v1 (기존 T3) | v2 (AI 비서) |
|---|---|---|
| 메인 화면 | 그달 사건·음악을 다 펼침 | **빈 기억 입력칸** + 옆에 AI 비서 |
| 정보 제공 | push — 시스템이 다 차림 | **pull — 사용자가 궁금한 걸 물음** |
| 주도권 | 시스템 | **사용자** |
| 읽기 부담 | 많음(관심 없어도 봐야) | 없음(필요할 때만 꺼냄) |
| AI 역할 | 없음(정적 표시) | **기억의 실마리를 꺼내주는 조수** |

**왜 더 나은가**
- 관심 없는 정보에 피로해지지 않는다 — 안 물으면 안 나온다.
- 능동적이다 — "이때 뭐가 유행했지?" 내가 궁금해서 물음 → 기억이 능동적으로 자극됨.
- 백지에서 시작하되, 막히면 비서가 도와줌.

## 2. 화면 구성

```
┌─────────────────────────────┬──────────────────────┐
│  내 기억 (메인, 좌)          │  AI 비서 (조수, 우)     │
│                             │                        │
│  "○○○○년 ○월,               │  추천 질문(칩):         │
│   당신의 이야기"             │  [나라가 떠들썩했던 일?] │
│  ┌───────────────────────┐  │  [이때 유행한 노래?]    │
│  │ (자유 작성)            │  │  [인기 드라마·영화?]    │
│  │                       │  │  [그 시절 유행?]        │
│  └───────────────────────┘  │                        │
│  🎤 음성 입력                │  💬 비서 답변 영역       │
│                             │  (DB or 검색 결과)      │
├─────────────────────────────┴──────────────────────┤
│        ← 이전 달 보기        │      다음 달 보기 →      │
└─────────────────────────────────────────────────────┘
```

- **좌(메인): 내 기억 입력칸** — v1의 MonthStory가 화면 주인공으로 승격. 텍스트 + 음성(T4 그대로).
- **우(조수): AI 비서** — 추천 질문 칩 + 답변 영역.
- **하단**: 거꾸로 여행 내비게이션(이전/다음 달). v1 그대로 유지.

## 3. AI 비서 — 핵심 동작

### 3.1 추천 질문 (백지의 공포 방지)
- 어르신 대상 → 빈 채팅창 대신 **예시 질문을 칩(버튼)으로** 깔아줌. 고르기만 하면 됨.
- 예시: "이때 나라가 떠들썩했던 일은?" / "이때 유행한 노래는?" / "그때 인기 드라마·영화는?" / "그 시절 유행은?" / "그때 물가나 살림은?"
- 익숙해지면 직접 타이핑도 가능(자유 질문).

### 3.2 답변 출처 — DB 우선 + 검색 보조 (역할 분담)

**핵심 원칙: 위험도에 따라 도구를 나눈다.**

| 질문 성격 | 출처 | 톤 |
|---|---|---|
| **big 이벤트** (정치·사회, 스포츠 등 전국민 사건) | **검증 DB** (MonthEvent) | 자신 있게 ("○월에 △△가 있었어요") |
| **취향** (드라마·영화·개인 유행·동네 등) | **인터넷 검색** | 조심스럽게 ("이런 게 있었던 것 같아요. 확인해보세요") |
| **음악** | DB 차트(있으면) + 검색 보조 | DB는 자신 있게, 보조는 조심스럽게 |

**왜 이렇게 나누나**
- **big 이벤트 = 틀리면 안 됨 + 검증 쉬움 + 양 적음** → DB가 맞다. AI가 틀리게 말하면 신뢰 붕괴.
- **취향 = 무한 + 사람마다 다름 + 틀려도 실마리면 됨** → 검색이 맞다. DB로 다 못 덮음.
- 목적은 백과사전이 아니라 **기억 자극**. 틀린 실마리여도 "아니 그건 아니고…" 하며 본인 기억이 살아나면 성공.

### 3.3 ⚠️ 검색 답변의 가드 (반드시 지킬 선)
- **환각 방지**: 검색 기반 답은 **"확실하지 않다"는 톤**을 항상 유지. 사실로 단정하지 않음. 가능하면 출처/시점을 함께.
- **저작권 가드 (v1과 동일)**:
  - 가사 ❌ / 앨범·가수 이미지 ❌ / 기사 문장 그대로 복제 ❌
  - 사실만, 우리 표현으로 요약. 음악은 곡명·아티스트·순위(사실)만.
  - 음악 재생은 유튜브 **검색 링크**만(임베드 ❌).
- **RAG 가드 (제품 핵심 원칙 유지)**: AI는 사용자의 기억을 *대신 창작하지 않는다.* 정보를 꺼내 실마리를 줄 뿐, "당신은 이랬을 거예요"라고 기억을 지어내지 않음.

### 3.4 비서 → 내 기억으로 연결
- 비서가 꺼내준 사건/노래를 보고 "아 맞다" → 사용자가 **왼쪽 기억칸에 직접 작성**.
- (선택) 비서가 답한 big 이벤트(DB 항목)를 "내 타임라인에 추가" 버튼으로 바로 keptEvent에 담을 수 있음 — v1의 "남기기"를 계승.

## 4. 거꾸로 여행 — 유지

- 현재(2026.5)부터 한 달씩 뒤로. v1의 내비게이션(이전/다음 달, EARLIEST/LATEST 가드) 그대로.
- 각 달마다: 좌측 기억칸 + 우측 비서. 비서의 답은 그 달(year/month) 기준.

## 5. 기존 구현(T3~T6)과의 관계 — 살릴 것 / 바꿀 것

### 살린다 (거의 그대로)
- **T6 저장 구조** — 한 달 = UserMemory N+1행(keptEvents + 월 회고). 그대로. 좌측 기억칸 = 월 회고, 비서로 담은 사건 = keptEvent.
- **T4 음성→AI 다듬기** — 좌측 기억칸에 그대로. 변경 없음.
- **T1·T2 데이터 모델·시드** — MonthEvent, ChartSong 그대로. big 이벤트 DB로 활용.
- **내비게이션**(거꾸로 여행, 경계 가드) — 그대로.
- **음악 카드(T5)** — 비서가 음악을 답할 때 이 카드 형식 재사용(시대색·순위·유튜브 링크).

### 바꾼다 (새로 만듦)
- **메인 화면 레이아웃** — "사건 다 펼치기"(v1 T3) → "기억칸(메인) + AI 비서(조수)". *완전 대체.*
- **AI 비서 컴포넌트 (신규)** — 추천 질문 칩 + 답변 영역 + DB/검색 분기 로직.
- **DB/검색 라우팅 (신규)** — 질문 성격 → big이벤트면 DB, 취향이면 검색. 검색 가드 포함.
- **기존 사건 펼치기(EventItem 그리드)** — 비활성. 비서 답변 안에서 "내 타임라인 추가"로 대체.

### 지우지 않는다 (검증 전)
- v1 화면 코드는 지우지 말고 비활성/보류. 새 화면으로 기본 진입. 새 방식 검증 후 별도 cleanup.

## 6. 구현 Phase 제안 (v2)

- **V1. AI 비서 백엔드** — 질문→답변 라우팅. big이벤트=DB 조회(기존 getMonthScreen 활용), 취향=검색(가드 적용). 검색 답엔 불확실성 톤 + 저작권 가드.
- **V2. 비서 UI** — 추천 질문 칩 + 답변 영역. 음악 답은 T5 카드 재사용.
- **V3. 새 메인 레이아웃** — 좌(기억칸=기존 MonthStory 승격) + 우(비서). 기존 사건 그리드 비활성.
- **V4. 비서→타임라인 연결** — 비서가 답한 DB 사건을 "내 타임라인 추가"로 keptEvent에 저장(T6 구조 재사용).
- **V5. 검증·정리** — 통합 테스트 후 v1 잔재 정리.

> 핵심 검증 질문: **"비서에게 물어보며 기억을 되살리는 게, 사건을 펼쳐 읽는 것보다 재미있는가?"**

## 7. 검색 도구 — 결정됨

**도구: Claude API 내장 웹 검색 도구 (`web_search`)**
- 이미 쓰는 Claude API 안에서 해결 — 별도 검색 서비스·키 불필요. `lib/ai.ts`에 도구만 추가.
- 검색 + 요약이 한 번에 (Claude가 검색 결과를 읽고 자연어로 답).
- 비용: 검색 1,000회당 $10 (=1회 1센트) + 요약 토큰(Haiku 저렴). 질문 1회 ≈ 1.5~2센트.
- **출처 인용**이 함께 나옴 → "확인해보세요 + 출처" 자연스러움.
- 보조: 웹 fetch 도구는 추가 비용 없음(특정 페이지 더 읽을 때).

**구현 전 체크**
- ⚠️ **Claude Console에서 웹 검색을 관리자 활성화**해야 작동(사용자가 콘솔에서 켜기).
- 검색 요약 모델은 **Haiku로 시작** → 답 품질(특히 환각) 보고 부족하면 Sonnet으로 상향. 실제 답변 보고 결정.

**가드 (시스템 프롬프트로 강제)**
- 검색 결과를 요약하되 **가사·기사 문장 그대로 복제 금지**, 사실만 우리 표현으로.
- **불확실성 톤** — 단정하지 말고 "이런 게 있었던 것 같아요, 확인해보세요". 출처 첨부.
- 음악 답은 곡명·아티스트·순위(사실)만. 이미지·임베드 ❌. 유튜브는 검색 링크만.

**토큰 차감 (잠정)**
- DB 조회 답변 = **무료**(우리 DB만 읽음).
- 검색 기반 답변 = **2~3토큰** 잠정(음성 다듬기 1~2토큰과 유사 급). 실제 비용 보고 확정.

## 8. 남은 열린 질문
- 검색 답 캐싱 — 시대 공통 질문(예: "1995년 유행")은 사용자 공통이라 캐시로 비용·속도 절약 가능? (개인정보 없는 일반 질문만)
- 추천 질문 칩 개수·문구 — 몇 개가 적당한지(너무 많으면 또 피로). 4~5개로 시작 권장.
