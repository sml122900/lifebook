# 2026-06-03 — v3 인생 연혁 사용성 시리즈 (v3.0+ ~ v3.5)

L1~L7 v3 피벗(어제) 후 통합 테스트로 발견한 결함들을 한 세션에 정리. 매달 부담 → 가로 시간축 전환은 끝났지만, 실제 사용자 시나리오(시니어가 흐릿한 기억으로 폼 채움 + 연혁 위 빈 공간 클릭)에서 새 문제가 잔뜩 떠올랐다.

총 5개 단계 — 모두 v3 코드 위에 *추가* 또는 *교체* 만, 새 모델 0(L1 정책 계승), 룸·반응·진척 코드 0줄 영향.

---

## 0. 통합 테스트로 드러난 1단계 결함

L7 끝낸 직후 본인 로그인으로 9개 카테고리 폼 채워보다 발견:

1. **건너뛰기 버그**: 3번 건너뛰고 4·5·6 답하고 7 건너뛰면 다음 미답으로 *다시 3번*. 의도는 "건너뛴 건 다시 안 나오게" 였는데 상태가 어디에도 안 남음.
2. **기간 카테고리에 끝 연도 없음**: 학교/직장/군대는 "입학~졸업" 처럼 양 끝이 의미 있는데 시작 한 점만 표시.
3. **나이 인지 보조 부재**: 시니어는 *연도* 못 떠올려도 *나이* 는 안다. 폼에 연도만 받으면서 나이 환산을 안 보여줘 추정이 어려움.

이 셋 + 카테고리 자체 개편(다음 단계) + 시각 통일(가로→세로) + 빈 공간 클릭 → 마이크로 시리즈로 정리.

## 2. v3.1 — 카테고리 개편 (10개)

통합 테스트로 본 흐름: "학교 한 줄에 초·중·고·대 다 담는 건 어색하고, 큰 이사·그 외는 인생 골격 질문으로 부담". 사용자 결정으로 10개 명확한 시점 카테고리로 재정의.

### 마이그레이션 2 (`20260602230155_v3_categories_overhaul`)

```sql
-- 매핑 불가 행 삭제
DELETE FROM "UserMemory" WHERE category IN ('RESIDENCE', 'OTHER');
-- skipped 셋 정리
UPDATE "User" SET "skippedLifeCategories" = array_remove(... 4종 모두 ...);

-- 새 enum + 의미 매핑
CREATE TYPE "LifeCategory_new" AS ENUM (
  'BIRTH','KINDERGARTEN','ELEMENTARY','MIDDLE','HIGH','UNIVERSITY',
  'MILITARY','WORK','RELATIONSHIP','FAMILY'
);
ALTER TABLE "UserMemory" ALTER COLUMN category TYPE "LifeCategory_new"
  USING (CASE category::text WHEN 'CHILDHOOD' THEN 'KINDERGARTEN'
                              WHEN 'SCHOOL' THEN 'ELEMENTARY'
                              ELSE category::text END)::"LifeCategory_new";
ALTER TABLE "User" ALTER COLUMN "skippedLifeCategories" TYPE "LifeCategory_new"[]
  USING "skippedLifeCategories"::text[]::"LifeCategory_new"[];
DROP TYPE "LifeCategory";
ALTER TYPE "LifeCategory_new" RENAME TO "LifeCategory";
```

### 새 카테고리 (순서)

| # | enum | 라벨 | 기간? |
|---|------|------|------|
| 1 | BIRTH | 출생 | 단일 |
| 2 | KINDERGARTEN | 어린이집·유치원 | 기간 |
| 3 | ELEMENTARY | 초등학교 | 기간 |
| 4 | MIDDLE | 중학교 | 기간 |
| 5 | HIGH | 고등학교 | 기간 |
| 6 | UNIVERSITY | 대학교 | 기간 |
| 7 | MILITARY | 군대 | 기간 |
| 8 | WORK | 첫 직장 | 기간 |
| 9 | RELATIONSHIP | 결혼 | 단일 |
| 10 | FAMILY | 자녀 | 단일 |

삭제: CHILDHOOD, SCHOOL(통합), RESIDENCE, OTHER

### 트러블슈팅: ALTER COLUMN USING 절의 서브쿼리 금지

`User.skippedLifeCategories` 변환에 처음엔 `USING ARRAY(SELECT ... FROM unnest(...))` 로 짰는데 Postgres 가 `cannot use subquery in transform expression` 으로 거부. 해결: text[] 경유로 직접 캐스트.

```sql
-- Before (실패)
ALTER TABLE "User" ALTER COLUMN "skippedLifeCategories" TYPE "LifeCategory_new"[]
  USING (ARRAY(SELECT c::text::"LifeCategory_new" FROM unnest("skippedLifeCategories") AS c));

-- After (성공) — 배열 전체를 text[] 로 캐스트한 뒤 enum[] 로 재캐스트
ALTER TABLE "User" ALTER COLUMN "skippedLifeCategories" TYPE "LifeCategory_new"[]
  USING "skippedLifeCategories"::text[]::"LifeCategory_new"[];
```

배열 안 모든 원소가 새 enum 에 존재해야 하므로 *이전 step* 에서 array_remove 로 옛 값 정리가 선행 조건.

### 코드 변경

- `lib/life-record/questions.ts` — `LIFE_QUESTIONS` 10개 (학교별 question/hint/placeholder 분리)
- `lib/life-events.ts` — `PERIOD_CATEGORIES` 7종 (학령기 5 + MILITARY + WORK)
- `lib/age.ts` — `schoolYearsForCategory(category, birthYear)` 신규 (KINDERGARTEN=만4~6, ELEMENTARY=만7~12, MIDDLE=만13~15, HIGH=만16~18, UNIVERSITY=만19~22)
- `/life-record/[category]/page.tsx` — SCHOOL 통합 힌트 → 각 학령 카테고리 단일 안내
- `EventForm` — `PERIOD_CATEGORIES` 미러 갱신, 기본 카테고리 `OTHER` → `BIRTH`
- 테스트 스크립트 4종 enum 참조 갱신

## 3. v3.2 — 세로 타임라인 (가로 → 세로)

이벤트 5개 넘어가면 가로 시간축에서 라벨이 겹쳤다. 데스크톱·모바일 모두 *세로* 로 통일.

### 레이아웃

**데스크톱 (sm+)** — 중앙 세로선 + 좌우 교차 카드 (짝수 우, 홀수 좌)
```
       ┌──────────┐           │
       │1972년 3월│───────────●  EXACT
       │ELEMENTARY│           │
       │입학 (만7) │          ┃  ← 기간 amber-500
       └──────────┘           ┃
                              ●           ┌──────────┐
                              ┃───────────│1978년쯤  │
                              │           │ELEMENTARY│  APPROX
                              │           │끝(만13)  │
```

**모바일 (sm-)** — 왼쪽 세로선 + 우측 카드만

### 기간 시각화

`computePeriodFlags()` 가 각 행에 `{topHalf, bottomHalf}` 부여:
- 시작 행: bottomHalf=true → 점 아래 절반 amber-500
- 끝 행: topHalf=true → 점 위 절반 amber-500
- 중간 행 (기간 안에 다른 이벤트가 끼는 경우): 양쪽 절반 amber-500 → 시선 끊김 없음

base 선(amber-100) 위에 강조선(amber-500) 덮어쓰기, 같은 `w-1` 굵기로 시각 일관.

### 라벨에 나이 곁들임

`birthYear` 있으면 when 줄 끝에 `(만 N세)` 작게.

## 4. v3.3 — 빈 공간 클릭으로 이야기 추가

연혁이 그려진 뒤 "이 시기에 떠올랐는데 어디서 추가하지?" 라는 흐름이 끊겼다. 빈 공간 클릭 + 점 옆 + 버튼 두 진입점.

### pointer-events 분리

```
ol.pointer-events-none
├── LineClickArea button.pointer-events-auto      ← 빈 영역 클릭
│     선 ±20px 폭 (w-10 + -translate-x-1/2)
└── li.pointer-events-none
      ├── EventCard.pointer-events-auto            ← 카드 (월별 이동)
      ├── 점 Link.pointer-events-auto              ← 점 (월별 이동)
      └── + 버튼 Link.pointer-events-auto          ← 추가
```

ol/li 자체엔 핸들러 없어 클릭 흡수 X — 점·카드·+버튼만 받고 그 외는 LineClickArea 로 통과.

### 연도 추정

`useRouter` + onClick:
```ts
const rect = e.currentTarget.getBoundingClientRect();
const ratio = clamp((e.clientY - rect.top) / rect.height, 0, 1);
const year = min === max ? min : Math.round(min + ratio * (max - min));
router.push(`/life-timeline/add?year=${year}&hint=1`);
```

`yearRange` 는 서버에서 `renderEvents.map(e => e.eventYear)` 의 min/max — 끝 점(`endYear` 미러) 도 포함.

### + 버튼

- 위치: `absolute left-full ml-2 z-20` — 점 우측 8px
- 크기: `h-10 w-10` (40×40, 시니어 친화)
- 색: `bg-amber-100` + `border-amber-400` + `text-amber-800`
- 노출: 데스크톱 `group-hover` + `group-focus-within`, 모바일 항상
- 동작: `/life-timeline/add?year=${eventYear}&hint=1`

### add 폼 prefill

- `searchParams.year` 검증(`1900 ≤ n ≤ next year`)
- `EventForm.defaultYear` prop 추가 — `yearText` 초기값 + `formMode` 강제 `"exact"`
- `hint=1` 이면 amber aside: "1989년쯤의 이야기를 추가해보세요. 연도는 바꾸실 수 있어요."

### TimelineView 클라이언트 전환

useRouter 필요해 `"use client"` 로. `calcAge` + LifeEvent type only import 라 안전. 번들 약간 증가하지만 컴포넌트 자체가 가벼움.

## 8. 결정·트레이드오프

- **endYear 컬럼 1개 vs 시작·끝 행 2개**: 한 행에 endYear 추가 → 룸·반응이 한 메모리 단위로 자연스럽게 작동 (시각화에서만 split). 두 행 방식은 반응이 어느 점에 달렸는지 모호.
- **건너뛰기 상태 저장처**: `User.skippedLifeCategories LifeCategory[]` native enum array (새 모델 0). 마이그레이션 한 줄.
- **카테고리 매핑 정책**: 사용자 결정 — `CHILDHOOD → KINDERGARTEN`, `SCHOOL → ELEMENTARY` 의미 매핑, `RESIDENCE/OTHER` 행은 삭제. L4 manage 에서 사용자가 다른 학령으로 옮길 수 있음.
- **세로 타임라인 클라이언트화**: `LineClickArea` 만 분리해도 됐지만 `TimelineView` 전체를 `"use client"` 로 — 작은 컴포넌트라 번들 영향 미미, 분리 비용 ↓.
- **카드 좌우 교차 + 모바일 한쪽**: 데스크톱은 양쪽 교차로 시간감 표현, 모바일은 좁아서 한쪽 — 두 layout 명시 분리(`hidden sm:block` / `sm:hidden`).
- **글로벌 위젯 위치**: 우측 하단 관습 따름. 사이드 패널 일부 겹침 알지만 일관성 우선.
- **/billing 유지**: 결제 UI 는 별도 페이지가 자연스러움(토스 콜백 등). `/account/tokens` 는 "매일 보는 곳", `/billing` 은 "가끔 충전".
- **사이드 패널 AttendanceMini 유지**: 사이드는 빠른 접근, 정식 페이지는 `/account/tokens` — 두 진입점 의미 다름.
