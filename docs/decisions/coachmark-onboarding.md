# 결정 — 코치마크 온보딩 (재사용 가능한 spotlight 엔진)

2026-06-27. 어르신·가족 첫 사용자가 화면을 바로 이해하도록 오버레이
코치마크를 도입하되, 화면마다 베끼지 않고 **엔진 1개를 재사용**한다.

## Problem

첫 사용자(특히 고령층)가 `/life-timeline`·포스터 흐름에 오면 버튼이 많아
"뭘 먼저 눌러야 하지" 하고 헤맨다. 카카오·배달앱식 첫 실행 가이드(어두운
배경 + 핵심 버튼 spotlight + 짧은 설명)가 필요하다. 단, 화면이 여럿이라
(메인 + 포스터 3화면) 화면마다 오버레이를 따로 만들면 유지보수가 깨진다.
"엔진은 한 번만, 화면은 데이터로" 가 목표.

제약: (1) 시니어 친화(큰 글씨·쉬운 말·한 번에 하나), (2) 첫 1회만 자동·
재실행 가능, (3) spotlight 가 타겟에 **정확히** 붙어야 함(어긋나면 오히려
혼란), (4) 사이드 패널·모바일 오버레이 같은 레이아웃 변수 흡수.

## Action

### 결정 1 — spotlight = 거대한 box-shadow 구멍

타겟 위치 박스에 `box-shadow: 0 0 0 9999px rgba(0,0,0,.62)` 를 줘 *주변만*
어둡게 한다. 마스크·클립패스·4분할 div 대신 한 요소로 딤 + 구멍을 동시에.
핵심 이점: **타겟의 z-index 와 무관** — 오버레이를 z-60 에 두면 사이드
패널(z-50) 안 버튼도 구멍으로 드러난다. 별도 portal·레이어 계산 0.

### 결정 2 — 단계는 순수 데이터(`CoachStep[]`), 화면은 `data-tour`

`lib/tours.ts` 에 `{target, title, desc, panel?}` 배열만 정의하고, 화면엔
`[data-tour="<target>"]` 속성을 단다. 엔진은 `querySelector` 로 찾아 측정.
없으면 그 단계는 skip(상태별로 안 보이는 버튼 견고). **새 화면 추가 = steps
배열 1개 + 속성 몇 개**, 엔진 0줄. 메인→포스터 3화면 확장이 이걸 실증.

### 결정 3 — 측정은 rAF 폴링 + ResizeObserver (정확도의 핵심)

"고정 지연 후 1회 측정" 은 어긋난다. 패널 슬라이드·스크롤 정착·비동기
렌더로 타겟이 *측정 후* 이동하기 때문. 그래서:
- 타겟을 즉시(`behavior:"instant"`) `scrollIntoView` 한 뒤, 위치가 **4프레임
  연속 안정될 때까지** 매 프레임 재측정(패널 단계는 슬라이드를 끝까지 잡으려
  최소 520ms 강제). 조기 정지가 모바일 토큰 카드를 슬라이드 전 off-screen
  위치(`left:1049`)에 고정시키던 버그의 원인이었다.
- 단계 전환 시 다음 타겟이 화면 밖이면 매 프레임 **재스크롤**해 ring 을
  다시 붙인다(spotlight 가 사라지고 배경만 남던 문제).
- `ResizeObserver(document.body)` 로 위 콘텐츠가 밀리는 늦은 layout shift 도
  추적. spotlight 위치 transition 은 제거(측정값에 정확히 스냅).

### 결정 4 — 말풍선 위치 일반화(여유 공간 기반)

타겟 위/아래 **실제 여유 공간**(말풍선 대략 높이 + 여백)을 계산해 배치:
둘 다 부족하면(상단 타겟·화면보다 큰 포스터 시안) 화면 위/아래 고정,
한쪽만 여유면 그쪽에. 상단 타겟에서 말풍선이 `top:-220` 으로 잘리던 버그를
일반 로직 하나로 해소(특수 케이스 `tall` 흡수). 모바일은 패널이 본문을
가려 항상 고정 모드 + 화살표 생략.

### 결정 5 — 저장은 컬럼 1개, 재실행은 이벤트

완료/건너뛰기 표시는 `User.completedTours String[]` 하나(투어별 행·컬럼
추가 X — id 만 누적). `markTourCompletedAction` 은 멱등·race-safe raw SQL
`array_append(array_remove(completedTours, $id), $id)`(skip 카테고리 패턴
재사용). 재실행은:
- CoachMarks 를 완료 사용자에게도 **항상 마운트**(이벤트 수신용), 자동
  시작은 미완료일 때만.
- 같은 페이지 = `START_TOUR_EVENT` dispatch(RSC 재렌더·쿼리·key 의존 0),
  다른 페이지 = `?tour=main` 이동 후 마운트 시 자동 시작.
- `begin()` 이 완료 플래그 리셋 + 패널 닫기 + URL 정리를 한 번에.

### 결정 6 — 단계별 사이드 패널 제어 + 모바일

콘텐츠 버튼 단계는 패널을 닫고, 패널 안 메뉴 단계는 연다(`SIDE_PANEL_EVENT`
커스텀 이벤트 → `SidePanelLayout`, localStorage 선호는 안 건드림). 모바일에서
패널이 본문을 가리는 문제를 단계 전환만으로 해결. 포스터 단계는 패널 무관
이지만 `begin()` 의 시작 시 패널 닫기로 모바일에서 포스터가 가려지는 것 차단.

## Result

- 신규: 엔진 `CoachMarks` 1 + 데이터 `lib/tours.ts` + 래퍼 `ScreenTour` 1 +
  서버 액션 1 + 컬럼 1(마이그 1, ADD COLUMN). 화면 적용은 `data-tour` 속성과
  `steps` 배열뿐.
- 재사용 실증: 메인 5단계 → 포스터 3화면(`poster-templates`/`select`/`view`)을
  **엔진 0줄 수정**으로 확장.
- tsc·lint 0(신규/수정 파일). 정적 검증까지 — 동작은 화면 확인 후속.

### 트레이드오프

- 첫 진입 자동은 기존 사용자에게도 1회 노출(빈 컬럼) — 의도된 소개로 수용.
- 도움말 버튼(좌하단)과 AI 비서 FAB(우하단) 두 floating 공존 — 늘면 정리.
- 포스터 시안은 화면보다 커 딤이 거의 안 보임 — 말풍선만 안내(허용).

### 일반화된 학습

1. **"엔진은 한 번, 화면은 데이터로"** — 오버레이 로직을 컴포넌트에 가두고
   화면은 `data-tour` 속성 + steps 배열만 노출하면, 새 화면 온보딩이 코드가
   아닌 데이터 추가가 된다(메인→포스터가 이를 증명).
2. **DOM 측정은 "한 번"이 아니라 "안정될 때까지"** — 트랜지션·비동기 렌더가
   있는 화면에서 단발 측정은 어긋난다. rAF 폴링 + ResizeObserver + 안정
   프레임 감지가 정확도를 만든다.
3. **재실행은 RSC/쿼리가 아니라 클라 이벤트로** — 항상 마운트 + 커스텀
   이벤트가 "같은 페이지 즉시 재시작" 을 RSC 재렌더 의존 없이 깔끔히 푼다.
4. **z-index 싸움은 box-shadow 구멍으로 우회** — 마스크 대신 거대한 그림자는
   타겟의 쌓임 맥락과 무관하게 어디서든 spotlight 를 만든다.
