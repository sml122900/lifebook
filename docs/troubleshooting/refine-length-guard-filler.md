# 문장 다듬기 길이 가드(0.8)가 군말 많은 글을 과차단 + 사투리 표준화 누출

2026-06-12. 문장 다듬기 Lv2(`lib/memory-refine.ts`) 실측 중 발견.

## 문제상황

Lv2 스펙대로 "결과 길이 80~120%" 를 프롬프트 + 서버 가드 양쪽에 넣고
군말 많은 표본으로 실측:

> 원문(123자): "어 그때가 그니까 인제 1988년이었는데 어 내가 인제 회사를
> 처음 들어갔거든. … 음 그래가 어머이가 억수로 좋아하시더라고."

결과 `status: no_change` — 다듬기가 아예 안 된 것처럼 보임. 두 번째
문제로, 모델 raw 출력에서 "어머이"→"어머니" 표준화(사투리 제거 금지
위반)도 발견.

## 시도한 것들

1. **모델 raw 출력 직접 확인** — `refineMemorySpelling` 을 거치지 않고
   같은 프롬프트로 `chat()` 직접 호출. 출력 82자 = **67%**. 다듬기 품질
   자체는 좋았다(군말 전부 제거, "타가지고"·"억수로" 보존, 반복 정리).
   즉 모델 실패가 아니라 **0.8 하한이 정상 결과를 차단**.
2. 원인 구조화 — 군말 비중이 큰 글일수록 제거만으로 길이가 크게 준다.
   "가장 다듬기가 필요한 글에서 no_change" 라는 모순. 하한 0.8 은
   책상에서는 합리적이었지만 도메인(입말 회상) 실측과 어긋남.
3. 선택지 비교 — (a) 하한 0.6 완화 (b) 하한 유지 + 프롬프트 수치 제거.
   (b)만으로는 모델이 길이를 맞추려고 군말을 덜 지우는 부작용 우려.

## 최종 해결법

- `LENGTH_RATIO_MIN` 0.8 → **0.6** (MAX 1.2 유지) — 군말 제거는 통과,
  요약(절반 이하)은 여전히 차단. 완화 이유를 상수 주석에 실측치(67%)와
  함께 명시.
- 프롬프트에서 "80~120%" 수치 삭제 → "요약하지 마라. 군말 제거 외에는
  문장이나 내용을 빼지 마라." (행동 지시가 수치보다 정확)
- 사투리 보강 — 예시 열거(어머이·어무이·억수로·~했당께·~카더라·그라모)
  + "군말(어, 음, 그니까, 인제, 막)과 사투리는 다르다 — 군말만 제거".
- 재실측: 94자(76%) `refined` 통과, "어머이" 보존 + 군말 전부 제거.

## 이력서 소재 한 줄

AI 텍스트 교정의 왜곡 방어 가드(길이 비율)를 도입하되, 실측 1건으로
임계값이 도메인(군말 많은 입말)을 과차단함을 발견해 근거와 함께 재조정
— "프롬프트 지시 + 서버 검증 쌍" 설계와 "가드 임계값은 실측으로" 원칙.
