## 1. V3 — 멀티턴 대화 + 답변 저장

### 멀티턴 (컨텍스트 답 우선)

기존 비서는 질문→답 단발. v3 핵심: 후속 질문은 **검색 안 하고 이전 답을
풀어 설명**하는 게 default. 토큰 절약 + 빠름.

`askAssistant(userId, q, year, month, prior?)`:
- `prior` 가 비어있지 않으면 → 컨텍스트 답 분기
- `chat(messages, system: contextSystemPrompt)` — web_search 없이 일반 호출
- 모델이 정확히 `[SEARCH]` 한 단어를 뱉으면 → 검색 폴백
- 그 외 → 컨텍스트 답 채택 (보통 1토큰 차감)

길이 가드:
- `MAX_PRIOR_TURNS = 8` (최근 8턴)
- `MAX_PRIOR_TEXT_CHARS = 600` (각 텍스트 cap)

컨텍스트 system prompt 가 처음엔 너무 보수적이라 거의 모든 후속에서
SEARCH 던짐 → 폴백 후 DB 분기 다시 들어가 같은 답이 두 번. 두 가지 보강:
1. 프롬프트 보강 — "이전 답에 등장한 항목을 풀어 설명하는 것은 환영.
   정말 새 사실이 필요할 때만 SEARCH"
2. 폴백 정책 — `[SEARCH]` 시 DB 건너뛰고 바로 검색 (DB 는 첫 답에 이미 줬음)

실측: "1번 자세히" → 컨텍스트 답으로 자연스럽게 풀어 설명, 1토큰 차감.

### 답변 저장

새 `createdVia="timemachine_assistant"` UserMemory 행 + content 에
AnswerSnapshot 의 JSON.

T6 와 분리:
- `saveTimemachineMonth` 의 `deleteMany` 필터는 `[event, month]` 만 →
  비서 행 절대 안 지워짐
- Phase 7 `ai_chat` 도 createdVia 가 달라 분리

UI: 탭 토글 `[채팅] [저장된 답변]`. 저장된 답은 재조회 시 토큰 0 으로
재렌더.

미픽스 (다음에 묶음): T1, B4~B7, C1, S2, S4, K1~K5.

### 검색 답 환각 줄이기 — system prompt 강화

검색용 system prompt 에 "**검색 결과를 다루는 원칙**" 섹션 4조항 추가:
1. 결과 빈약하면 인정 — 억지로 채우지 않기
2. 검색 결과에 명확히 등장하는 사실만 단정 + **일반 상식/사전 학습으로 빈칸 메우지 말 것**
3. 미확정 내용은 답 안에 "확실하지 않다" 표시
4. 모르면 "정확한 자료를 못 찾았어요" — 추측보다 낫다

기존 가드(가사·기사 복제, 단정 톤, 이미지·임베드, 시니어 친화) 모두 유지.
모델은 그대로 Haiku.

## 3. V4 — 답의 깊이 (3단 모델 라우팅)

사용자 라벨: **간단히 / 자세히 / 가장 정확하게** (모델 이름 절대 노출 X).
검색 답에만 적용. DB 답은 무료 유지.

| depth | model | base × multiplier | 검색 답 평균 |
|---|---|---|---|
| simple | claude-haiku-4-5 | 1× | 10토큰 |
| detailed | claude-sonnet-4-6 | 3× | 30토큰 |
| precise | claude-opus-4-7 | 5× | 50토큰 |

### 토큰 정책 — 호출부 surcharge 로 차이 흡수

기존 `tokensFromUsage` + `chargeOneShot.surcharge` 무변경. 호출부에서:
```ts
const base = tokensFromUsage(in, out);
const surcharge = base * (MULTIPLIER[depth] - 1) + WEB_SEARCH_SURCHARGE;
chargeOneShot(userId, in, out, `..._${depth}`, undefined, surcharge);
```

Haiku 일 땐 `multiplier=1` → `surcharge = 0 + 1 = 1` → 기존 정책과 동일
(회귀 0). Sonnet/Opus 만 추가 가산.

### Opus 4.7 — temperature 거부

`temperature is deprecated for this model` 400 에러. Opus 4.7 은 reasoning
모델이라 temperature 안 받음. `lib/ai.ts` 에 `supportsTemperature(model)`
가드 → Opus 면 조건부 미전송.

### UI

깊이 토글 (라디오 3개) 상단. 각 옵션에 추정 토큰 미리 표시 ("약 10토큰"
"약 30토큰" "약 50토큰"). 답 카드에 `[간단히 답]` / `[자세히 답]` 배지.
후속 대화는 같은 깊이 자동 이어감 (state).

### 검증 (`db/test-assistant-depth.ts`)

| 시나리오 | 결과 |
|---|---|
| (a) 3 깊이가 각각 다른 모델 호출 | ledger 에 `_simple`/`_detailed`/`_precise` 분리, 콘솔 model= 로그로 확인 |
| (b) 토큰 비례 차등 | simple=10, detailed=43, precise=56 (실측 in 토큰 차이로 ratio 1.3x — multiplier 1.67x 미만은 검색 비결정성) |
| (c) DB 답 무료 유지 | 3 깊이 모두 tokensSpent=0 |
| (d) 컨텍스트도 이어감 | simple=1, precise=5 (정확히 1×, 5×) |

## 4. 출석체크 (시각 개편 포함)

### 데이터 모델 — UserAttendance

```prisma
model UserAttendance {
  id         String @id @default(cuid())
  userId     String
  date       String  // KST "YYYY-MM-DD"
  streak     Int     // 이 날 출석 후 누적 연속일
  bonusToken Int     @default(0)  // 0 또는 30
  createdAt  DateTime @default(now())
  @@unique([userId, date])
}
```

`wallet`/`TokenTransaction` 무변경. 새 ledger reason 두 종.

### KST 처리

```ts
function kstDateString(d = new Date()) {
  return new Date(d.getTime() + 9 * 3600 * 1000).toISOString().slice(0, 10);
}
```

서버 시간 무관, timezone 라이브러리 의존 0.

### Race-safe — `@@unique([userId, date])` 가 단일 결정자

```
$transaction {
  UserAttendance.create   ← P2002 위반 가능
  wallet.balance += credit
  TokenTransaction.create × 1 또는 2
}
catch (code === 'P2002') → "이미 출석" 분기
```

동시 두 요청에도 하나만 성공, 나머지는 친화적 안내.

### 정책

- 매일 5토큰
- 7의 배수 streak (7, 14, 21…) 마다 +30 보너스 (계속 누적 정책)
- 끊기면 streak=1 리셋
- 비난·압박 표현 0 — "오늘도 와주셨네요"

### 시각 (AttendanceCard) — 동그라미 7개

```
출석체크                        3일 연속 출석중
 ✓   ✓   ✓   4   5   6   7
amber amber amber zinc zinc zinc zinc        +30
          이번 사이클 3/7일
┌─ 이번 사이클 보상 ─────────────┐
│ 매일 출석           5토큰       │
│ 7일 연속 보너스    +30토큰      │
│ 지금까지 받은 토큰  15토큰       │
└────────────────────────────────┘
[ 오늘 출석체크하기 (5토큰) ]
```

streak=6 일 땐 "오늘 출석체크하기 (5토큰 + 보너스 30토큰!)" 예고. 받은
후엔 회색 비활성 "오늘 출석 완료!".

사이드 패널 미니: 16px 작은 동그라미 7개 + N/7 텍스트.

### 검증 (`db/test-attendance.ts`)

(a) 같은 날 두 번 / (b)(c) 7일 연속 + 보너스 / (d) 동시 요청 race-safe /
(e) 거른 후 리셋 / (f) 14일째 또 보너스 / (g) 기존 chargeOneShot 무영향
— **모두 통과**.
