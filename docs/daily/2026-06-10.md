## 2. 화면 테스트에서 드러난 SSR 버그 2건

C 가 `PlaceSearchInput` 을 **`/life-timeline`(TimelineView) 에 처음** 끌어오면서, 그동안 다른 라우트에만 있어 안 보이던 잠재 버그 2건이 메인 페이지에서 터졌다.

### 버그① — `"use server"` 파일이 number 를 export

```
Error: A "use server" file can only export async functions, found number.
```

`app/era/actions.ts`(맨 위 `"use server"`)가 `export const ERA_MEMORY_LIMIT = ERA_MEMORY_MAX_LENGTH`(=500) 를 내보내고 있었다(E3 의 알려진 후속). 평소엔 통과하다, 장소 저장 액션(`updatePhotoPlaceAction`)이 같은 `/life-timeline` 액션 번들에 합류 → 번들 전체 **재검증** → number export 가 걸림.

- **막힌 곳**: 클라(`EraMemoryEditor`)는 이 숫자가 필요한데 `lib/era-stash`(prisma 의존, 서버 only)를 import 못 함 → 그동안 "use server" 액션 파일을 통로로 우회하고 있었다.
- **해결**: prisma 의존 없는 순수 모듈 **`lib/era-constants.ts`** 신설(= `place-types.ts` 패턴). 서버는 `lib/era-stash` 가 import+재노출(기존 `@/lib/era-stash` 호출자 호환), 클라는 직접 import. CLAUDE.md 의 "ERA_MEMORY 상수 두 곳 정의 통일" 후속도 동시 해결.

### 버그② — `window is not defined` (콜드 SSR 첫 요청 500)

```
ReferenceError: window is not defined
  at module evaluation (app/components/maps/GoogleMap.tsx:3:1)  // @googlemaps/js-api-loader
```

`@googlemaps/js-api-loader` 의 dist 가 **모듈 최상단**에서 `const trustedTypes = window.trustedTypes`(line 47) 를 평가 → 서버 모듈 평가 시 throw. 클라 컴포넌트라도 Next 는 SSR 위해 서버에서 모듈 평가. C 전엔 `/life-timeline` 이 이 체인을 안 끌어왔는데 이제 메인이 노출(`/add`·`/photos` 는 이미 들고 있어 동일 위험을 갖고 있었음).

- **증상 패턴**: 콜드 컴파일 직후 첫 요청만 500, 워밍 후 200 — 그래서 "가끔 깨짐"으로 보였다.
- **해결**: `PlaceSearchInput` 에서 `PlaceMap` 을 **`next/dynamic({ ssr: false })`** 로 로드. 지도는 순수 인터랙션(SEO 가치 X)이라 SSR 제외가 정답. `/add`·`/photos` 도 같이 단단해짐.
- **증거**: 픽스 후 로그에 `✓ Compiled` 직후 첫 `GET /life-timeline` 이 **200**(이전엔 매 콜드 컴파일 첫 요청 500).

## 4. 교훈

- **"use server" 파일은 number 도 못 내보낸다** — 클래스만이 아니다(use-server-class-export.md 재발). 클라가 서버 전용 모듈의 상수를 필요로 할 때 액션 파일을 통로로 쓰면 잠재 폭탄. 순수 상수 모듈이 단일 진실 원천.
- **잠재 버그는 "새 진입점"에서 터진다** — 두 버그 다 코드는 그대로였고, C 가 `PlaceSearchInput`/`updatePhotoPlaceAction` 을 메인 페이지·메인 액션 번들에 *처음 합류*시키며 표면화. 모듈 그래프에 새 노드를 더할 때 재검증이 도는 걸 염두.
- **브라우저 전용 라이브러리는 `ssr:false`** — 모듈 최상단 window 접근은 SSR 에서 무조건 터진다. 콜드-only 500 은 "가끔"이 아니라 "항상(워밍 전)" — 메인 동선에 들어오면 반드시 잡아야.

# 2026-06-10 (세션 2) — 사진 6: EXIF 날짜·GPS strip·대량 업로드·사건 첨부/빼기 + 코드리뷰

## 6. 교훈

- **순수 함수 경계로 테스트가 산다** — `stripGps`/`extractPhotoDate` 는 브라우저 API(FileReader) 라 node 테스트 불가지만, `movePhotoToMemory` 는 Storage 안 건드리는 순수 DB 라 17 시나리오 자동 검증. 클라 전용은 grep+빌드+사용자 실측, DB 로직은 tsx 테스트로 가른다
- **"모든 경로"를 grep 으로 못 박는다** — GPS strip 누락(H1)은 `append("file"` grep 한 줄로 4경로 중 1곳이 raw 인 걸 잡았다. 같은 정책이 여러 진입로에 흩어지면 grep 가 회귀 안전망
- **빼기≠삭제** — 어르신은 사진을 잃으면 회복 못 함. "사건에서 빼기"를 `deletePhotoOwned`(삭제) 대신 `movePhotoToMemory`(독립 복귀)로 둬 데이터 보존. 같은 "빼기" 단어라도 도메인이 다르면 연산이 다르다
