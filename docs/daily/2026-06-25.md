# 2026-06-25 작업 일지 — 장소 1:N (사건에 여러 장소)

## 배경

한 인생 사건(UserMemory)에 장소를 **단 1곳**만 붙일 수 있었다(평면 5컬럼 `placeName/placeAddress/lat/lng/placeSource`). 부모님 요청 — "한 사건에 여러 장소"(예: 신혼여행지 여러 곳, 직장이 옮겨다닌 여러 사무실). 1:1 → 1:N 전환을 H1~H5 단계로 나눠 안전하게(롤백·호환 보존) 진행.

전환 원칙: **추가만, 제거는 나중(H6)**. 기존 5컬럼은 호환·롤백용으로 H5까지 유지. 각 단계 `tsc --noEmit` 통과 확인.

## H1 — 스키마 + 백필 (마이그 2건, 성민이 migrate deploy)

- 새 모델 `MemoryPlace` (id/memoryId FK Cascade·index/placeName(NOT NULL)/placeAddress/lat/lng/placeSource/sortOrder/createdAt). `UserMemory.places MemoryPlace[]` 역참조 1줄.
- 마이그 `20260624000000_memory_place_table` — CREATE TABLE.
- 마이그 `20260624000001_backfill_memory_place` — 기존 UserMemory 중 placeName 있는 행 → MemoryPlace 1건씩(sortOrder=0). **멱등**: `NOT EXISTS` 가드로 재실행 안전, `gen_random_uuid()::text` id.
- ★ 5컬럼 삭제 X. `prisma migrate deploy` 적용 완료(성민이 실행, 44 마이그).

## H2 — 서버 읽기 (places[] 반환)

- `lib/life-events.ts` 3개 읽기 함수(`getLifeEvents`·`getLifeEventForCategory`·`getLifeEventById`): select에 `places` 관계 include(sortOrder ASC), 반환 타입에 `places: PlaceInfo[]` 추가. 단일 `place`도 당분간 유지.
- `lib/photos.ts` `listUserPhotos`·`listMemoryPhotos`: `UserPhoto.places[]` 추가(memory.places join).
- `lib/people.ts` `listEventsByPerson`(수동 LifeEvent 매핑): tsc 강제로 places select+map 추가.
- 빈 배열 = 장소 없음. 단일 place 소비처(TimelineView 등)는 무수정 → 동작 변화 0.

## H3 — 서버 쓰기 (MemoryPlace 기록)

- `LifeRecordInput.places?` 추가. 헬퍼 `normalizePlaces`(places[] 우선·없으면 [place] 흡수·`validatePlace`로 거른 valid + 5컬럼용 primary).
- `createLifeEvent`/`upsertLifeEvent`(create): `userMemory.create`에 중첩 `places: { create }`(암묵 트랜잭션).
- `updateLifeEvent`/`upsertLifeEvent`(update): `$transaction([update, deleteMany, createMany])` — 기존 싹 지우고 재생성(원자적·부분실패 전체 롤백).
- `lib/photos.ts` `buildPhotoMemoryData`에 중첩 places, `updatePhotoMemoryPlace`(나중 H5에서 배열화) 트랜잭션.
- ★ 5컬럼은 `primary`로 계속 write(호환).
- companion 추출은 UserMemory에 place 안 씀(장소=별도 `Person subjectType=location`) → 무관. 온보딩 사건화·G2는 `createLifeEvent` place 없이 호출 → 자동 처리.
- **보안 픽스**: `updatePhotoMemoryPlace` 트랜잭션화 시 MemoryPlace엔 userId 없어 deleteMany가 소유 검증 불가 → 남의 memoryId로 피해자 장소 삭제 가능. 소유 확인을 트랜잭션 *앞*에 두어 차단(updateLifeEvent `current` 패턴과 동일).

## H4 — UI 표시 (배열 렌더)

- `TimelineView.tsx` `PlacePreview`: 단일 → `places[]` 순회. placeName 있는 것만, **세로 한 줄씩 칩 나열**(어르신 친화 구분). 단일 칩은 `PlaceChip`로 추출(externalMapHref source 분기 유지). 빈 배열 → `return null`(빈 핀 X).
- 호출부 3곳(`e.place`→`e.places`) + `PhotoCard` `localPlace`→`localPlaces`.
- `poster/page.tsx`: 출생지 라벨 `birth.places.find(p=>p.placeName)`.
- `DraftReview`/`DraftLocationCard`는 Person(location) draft 입력이라 무관, people/[id]·/photos는 place 미표시 → 변경 불필요.

## H5 — 다중 입력 UI (핵심)

- **새 `app/components/PlacesEditor.tsx`** — `value: PlaceInfo[]`/`onChange` 배열 관리. 한 개 추가는 기존 `PlaceSearchInput` 재사용(선택 완료 시 append, "선택 안 함"=취소). 고른 장소 카드 + 큰 [제거] 버튼, "장소 추가"/"장소 더 추가"/"추가 취소".
- `EventForm`·`CategoryForm`: place 단일 state → `places[]`, PlaceSearchInput → PlacesEditor.
- `actions.ts`(life-timeline/life-record): `places?` 입력 추가, validate가 각 항목 검증 후 헬퍼에 `places` 전달. 단일 `place?`는 호환 유지.
- `PhotoPlaceModal`: 단일 → 다중(기본 0~1). `updatePhotoMemoryPlace` → 배열형 `updatePhotoMemoryPlaces`, `updatePhotoPlaceAction(memoryId, rawPlaces[])`.
- 업로드 폼(PhotosUploadForm)은 단일 그대로(업로드 시 1곳이 흔함).
- prefill: 편집 진입 시 sortOrder 순 places[] → 카드 N개. 0개 저장 OK / 다중 저장·재편집 / 제거 정상.
- F5 `PlaceableMultiItemWidget`(온보딩 LifeProfile Json 저장)은 별개 흐름이라 무수정.
